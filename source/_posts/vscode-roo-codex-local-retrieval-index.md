这篇文章的目标不是再讲一遍某个单独配置项怎么改，而是把本机上 VS Code、Roo、Codex、本地 MCP、skills 与经验镜像之间的关系整理成一份结构索引，方便以后排查“到底谁在读谁、谁在调用谁、谁只是磁盘上存在但没有真正进入运行链”。

## 1. 先说结论

当前这台机器上的本地检索与能力接入，不是一个单点配置，而是一组分层结构：

- VS Code 提供扩展宿主与运行会话
- Roo 通过扩展安装目录、私有持久化目录和活动 MCP 配置参与运行
- Codex 通过本地配置文件、skills 目录和本地知识镜像参与运行
- `experience-manager` 已经进入 Roo 当前 MCP 结构
- `qmd` 这一类本地检索能力更像 Codex 侧的 skill 约束与知识镜像入口

所以以后遇到“为什么没查到”“为什么改了不生效”“为什么磁盘上明明有能力却没被调用”这类问题，第一反应应该是先分层，而不是把所有东西混成一个整体。

## 2. 整体结构总览

从高层看，当前结构可以这样理解：

```text
VS Code
├─ Roo Code 扩展
│  ├─ 扩展安装目录
│  ├─ 扩展私有持久化目录
│  └─ 活动 MCP 配置
├─ Codex 本地配置
│  ├─ auth / config
│  ├─ skills 目录
│  └─ 本地知识镜像 / 检索 skill
└─ 本地能力仓
   ├─ MCP servers
   ├─ skills
   └─ 经验镜像与包装器
```

这张图最重要的不是记住目录名，而是记住职责：

- 扩展安装目录负责“扩展本体”
- 私有持久化目录负责“当前用户级状态与配置投影”
- 本地技能目录负责“行为约束与检索入口”
- 经验镜像负责“本地可查知识”

## 3. Roo 这边到底分哪几层

### 3.1 VS Code 中 Roo Code 扩展安装目录

这里承载 Roo 扩展本体。

需要注意的是：

- 它是扩展代码所在位置
- 但不是所有运行态问题都应该直接去这里找答案
- 如果这里出问题，往往属于扩展 bundle、本体文件或安装层的问题

也就是说，安装目录重要，但它不等于当前活动配置真源。

### 3.2 Roo Code 的 VS Code 持久化目录

这一层更接近当前用户维度的真实状态。

它通常会承载：

- 当前扩展私有设置
- 持久化存储
- MCP 配置投影
- 某些运行时状态

排查时如果只改仓库里的示例文件、不看这里，最容易出现“磁盘改了但当前运行态没切过去”的错觉。

### 3.3 Roo 本地能力仓

除了扩展本体和私有存储之外，本机还存在更宽一层的本地能力仓，用来承载：

- MCP 服务相关文件
- 本地技能
- 包装器/探针式能力
- 与经验镜像相关的外围资源

这层的重点是：

**能力存在于磁盘，不等于已经进入当前运行链。**

## 4. 当前 `experience-manager` 的定位

### 4.1 `experience-manager` 已接入 Roo MCP 配置

这是当前已确认的事实之一。

也就是说，它不是只存在于磁盘上的“可能能力”，而是已经在 Roo 的 MCP 配置结构里出现过。

### 4.2 `experience-manager` 在 Roo 中的定位

它更像经验层能力的接入点，而不是普通的静态文档目录。

从实际用途看，它承担的是：

- 经验访问入口
- 能力包装层
- 被上层 skill 或模式调用的桥接点

### 4.3 Roo 当前 `experience-manager` 更像包装器/探针

从实际观察结果看，当前它更接近“包装器/探针”角色，而不是完整自治的知识系统。

也就是说，它更像：

- 一个接入点
- 一个桥
- 一个把经验查询流程带入当前运行链的组件

### 4.4 Roo 现有 skill 会主动调用 `experience-manager`

这意味着它不是“挂在那里但没人用”的死配置，而是和现有 skill 存在真实调用关系。

因此以后如果要判断它是否有用，不能只看目录是否存在，而要看：

- 当前 skill 是否会调它
- 调用链是否仍然成立
- 当前运行态是否已经 reload 到最新配置

## 5. `qmd` 和自定义模式这边现在是什么状态

### 5.1 当前没有发现 `.roo` 侧的 `qmd` skill

至少在目前能确认的范围里，没有直接证据证明 Roo 侧已经有一个明确的 `qmd` skill 接入点。

这意味着：

- 不要想当然地把 Codex 侧检索能力当成 Roo 默认也有
- Roo 与 Codex 在本地检索链路上并不完全对称

### 5.2 当前 `custom_modes.yaml` 为空

这也是一个很重要的现状判断。

它说明当前机器上的 Roo 自定义模式层，并没有额外复杂的已落地模式编排在生效。

这既是简化点，也是风险点：

- 简化点在于当前行为更容易追踪
- 风险点在于很多“检索优先”之类的规则可能还没有被结构化固化下来

## 6. Codex 这边的本地技能与检索结构

### 6.1 Codex 本地技能目录

Codex 一侧存在自己的本地技能目录，这一层应被视为独立事实源。

因为它决定：

- 某些行为约束是否存在
- 某些本地检索逻辑是否会被优先触发
- 某些知识镜像是否会被接入使用

### 6.2 `qmd` skill 已经明确写了“先查再答”

这条信息很关键。

它说明 Codex 侧并不是“希望能检索”，而是已经在 skill 层明确表达出一种行为顺序：

**先查，再答。**

这意味着本地检索不只是一个资源层能力，也是提示词/技能层行为约束的一部分。

### 6.3 `qmd` 指向的经验镜像路径

`qmd` 并不是凭空工作，它需要指向本地经验镜像路径。

这说明检索效果依赖两个前提：

- skill 本身是否存在并被触发
- skill 指向的镜像内容是否真实、最新、可访问

### 6.4 刷新经验镜像的入口

经验镜像不是一劳永逸的静态真理。

如果镜像不刷新，再好的 skill 也只会查到过时内容。

所以这层后续要特别注意：

- 镜像更新入口在哪里
- 更新后谁来触发重载或重新索引
- Roo 和 Codex 是否共用同一批经验事实源

## 7. 功能分工表

为了避免概念混淆，可以把三者职责简单收束如下：

- VS Code：扩展宿主、窗口会话、用户配置承载
- Roo：扩展运行态、私有存储、活动 MCP 接入、模式/提示词运行链
- Codex：本地配置、skills、检索优先策略、本地知识镜像接入
- `experience-manager`：更偏包装器/探针式经验接入
- `qmd`：更偏 skill 层的检索优先入口

## 8. 当前机器上的现状判断

### 8.1 已确定存在

目前可以明确的内容包括：

- Roo 扩展安装目录与持久化目录是分层存在的
- Roo 当前 MCP 配置里存在 `experience-manager`
- Codex 存在本地技能目录
- `qmd` skill 已明确表达“先查再答”的思路
- 本地经验镜像链路是实际设计的一部分

### 8.2 还没有直接证据证明的内容

当前仍不能随意断言的内容包括：

- Roo 侧是否已有和 Codex 等价的 `qmd` skill
- 所有本地能力是否都已进入当前活动运行链
- 当前自定义模式层是否已正式承载检索优先逻辑

也就是说，后续任何判断都仍应以实际配置、活动会话和调用链为准。

## 9. 以后如果要继续改，优先改哪里

### 9.1 第一优先：Roo 的模式与提示词入口

因为这决定 Roo 运行时如何组织行为。

如果希望让 Roo 也稳定遵守“先查再答”，最优先考虑的应该是模式与提示词入口，而不是先去到处散改外围目录。

### 9.2 第二优先：在 `.roo\skills` 中补一个通用检索优先 skill

这样可以把经验性的操作原则升级成可复用规则，而不是继续依赖人脑记忆。

### 9.3 第三优先：保持 `experience-manager` 与 `qmd` 镜像同步

否则很容易出现：

- Roo 查的是一套经验镜像
- Codex 查的是另一套镜像
- 两边对“本机事实”的理解逐步漂移

## 10. 最后给自己留一条操作原则

如果以后再改本机这套 VS Code、Roo、Codex、本地检索链路，最重要的原则就是：

**先区分扩展本体、私有存储、活动配置、技能目录和经验镜像，再判断谁是真正事实源，不要看到磁盘上有文件就默认它已经进入当前运行链。**
