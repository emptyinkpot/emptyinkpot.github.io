这篇文档把几类原本分散的材料收口到一起：

- 顶部导航栏和共享导航的历史经验
- 文件管理、原生界面、主 UI 的入口分流经验
- 启动链、配置源、浏览器入口的复盘
- 前端页面新增、路由注册、验证与排障规则
- 后续模块化改造对导航和页面结构的正式约束

目标不是保留所有历史细节，而是提炼出一份**以后还能直接照着做**的操作指南。

## 1. 先分清“现行基线”和“长期目标”

过去这些资料里，最容易让人混乱的地方是：有些写的是**当前为了稳定运行而采用的过渡方案**，有些写的是**最终希望演进到的正式架构**。

这份文档明确按两层口径来写：

- **现行可运行基线**：今天这套 OpenClaw 在本机上怎样最稳地跑起来、怎样最稳地改导航和入口
- **长期正式目标**：后续模块化完成后，页面、导航、插件和运行态应该怎样收敛

如果这两层不分开，后面就会反复出现一种问题：

> 当前还在用共享导航 HTML 过渡，但文档又要求以后导航由 `module.json` 生成，结果维护者不知道今天该改哪一处。

## 2. 现行可运行基线

截至当前，这套 OpenClaw 的**实际有效前端与入口基线**应按下面理解。

### 2.1 当前真实启动链

用户日常打开 OpenClaw UI，应以这一条链路为准：

1. 点击桌面快捷方式
2. 调用 `start-openclaw-admin.ps1`
3. 转到 `start-openclaw.bat`
4. 读取当前真实配置源
5. 调用 `openclaw-gateway-run.ps1`
6. 启动或复用网关
7. 等待 `/health` 与 UI 页面同时就绪
8. 浏览器打开：

```text
http://127.0.0.1:<port>/control-ui-custom/index.html#token=...&gatewayUrl=ws://127.0.0.1:<port>
```

结论只有一个：

> **当前浏览器主入口是 `control-ui-custom/index.html#...`，不是随便哪个 HTML。**

### 2.2 当前真实配置源

当前真实生效的配置文件应按下面判断：

- 如果 `E:\My Project\Auto\.local\openclaw\openclaw.json` 不存在
- 那么真实配置源就是 `E:\My Project\Auto\projects\openclaw.json`

在当前状态下，后续操作默认应先检查 `projects\openclaw.json`，不要想当然去改 `.local`。

### 2.3 当前 Control UI 根目录

当前 UI 根目录基线是：

```text
gateway.controlUi.root = control-ui-custom
```

也就是说，当前 Control UI 是从 `control-ui-custom` 提供的。

### 2.4 当前共享导航运行方式

当前运行态里，导航仍然依赖共享导航注入机制。

现行过渡做法是：

- 页面最终读取 `/shared/nav-bar.html`
- `nav-bar-behavior.js` 负责注入与 active 状态修正
- 任何镜像、副本、缓存版本，都不能再被误当成唯一真源

这是一套**过渡期仍可运行**的机制，不等于最终正式架构。

## 3. 长期正式目标

长期目标应以模块化架构为准，而不是继续长期手改共享 HTML。

正式目标是：

- 页面归位到各模块 `frontend/pages`
- 路由归位到模块自己的声明或统一注册点
- 导航由 `module.json` 等正式声明生成
- 不再长期手改多份 `nav-bar.html`
- 不再保留 `extensions/public/control-ui-custom` 这类镜像副本

这意味着：

> **共享导航 HTML 目前可以作为过渡运行方案继续使用，但不能再当作未来长期架构的最终形态。**

## 4. 导航管理的现行操作规则

### 4.1 只允许一个真正的共享来源

导航统一管理最重要的不是“文件名一样”，而是：

> **所有页面最终都必须读取同一份共享来源。**

因此当前应遵守：

- 先识别哪一份是当前实际被页面读取的共享导航源
- 只改唯一真源，不改多个副本
- 修改后必须同步到真实 served 路径
- 再通过浏览器实际 DOM 验证是否生效

### 4.2 不要只看源码文件

过去导航乱码、导航内容不更新，很多次都不是源码文件本身的问题，而是：

- served 副本没更新
- 浏览器缓存没刷新
- 实际读取的不是你刚改的那份文件

所以排障顺序必须是：

1. 先看浏览器真实 DOM
2. 再看实际请求到的 `/shared/nav-bar.html`
3. 再看缓存版本号或静态资源版本
4. 最后才回头看源码

### 4.3 当前新增导航项的做法

如果当前运行态还依赖共享导航 HTML，那么新增导航项时应：

1. 先确定这个入口属于哪个模块
2. 先确定它是不是正式长期入口
3. 只在唯一共享来源里加一次
4. 同步 active 规则
5. 验证它最终落到的是正确路由，而不是旧别名

也就是说，**不能在多份 `nav-bar.html` 里各改各的**。

## 5. 页面入口与路由分流规则

这部分是过去最容易反复出错的地方。

### 5.1 主入口、原生入口、文件管理入口必须拆清

过去出过的很多问题，本质都是不同入口被混成了一类。

当前应明确区分：

- 主 UI：`/control-ui-custom/index.html`
- 兼容/过渡 bootstrap 页：`/control-ui-custom/launch.html`
- 文件管理：`/file-manager`
- 自动化：`/automation`
- 小说管理：`/novel`
- 经验中心：`/experience`

原则是：

- 不同功能必须落到不同路由
- 不能都回到根路径 `/`
- 不能让文件管理和原生界面共用一个“模糊入口”

### 5.2 `launch.html` 的当前定位

历史上 `launch.html` 承担过 token 预热和跳转职责，但**当前日常主入口已经不是它**。

现行口径应改成：

- 启动器日常打开的是 `index.html#token=...&gatewayUrl=...`
- `launch.html` 只保留为兼容、排障或过渡链路
- 后续如果 `index.html` 已稳定承担 bootstrap，就不要再把 `launch.html` 当主入口宣传

### 5.3 顶部导航里的入口必须是正式入口

顶部导航不应该继续保留这些旧入口或错误入口：

- `/auto.html`
- `/automation.html`
- `/publish.html`
- `/experience.html`
- 依赖根路径 `/` 的隐式跳转
- 已被移除但仍偶尔复活的 `/project-structure.html`

现行正式入口应优先使用：

- `/automation`
- `/novel`
- `/experience`
- `/file-manager`
- `/control-ui-custom/index.html`

## 6. 前端页面新增或改造的标准流程

以后新增一个页面、标签页、导航入口或工具页，建议固定按这个顺序执行。

### 6.1 先找模块归属

先回答两个问题：

- 这个页面属于哪个模块
- 这个页面是不是应该作为全局导航一级入口暴露

如果模块归属都没定，就不要先去改导航。

### 6.2 页面位置要对

长期正式结构里，页面应进入模块自己的：

```text
frontend/pages/
```

不要继续往模块外、镜像目录、`public-copy`、`_fixed`、`_new` 这种补丁目录里塞页面。

### 6.3 路由注册要对

如果当前模块还在旧插件注册阶段，那么新增页面时至少要保证：

- 页面文件存在
- 路由映射已注册
- 对应 routes 声明已注册

不能只改 `pageMap` 或只改某一层声明。

如果当前模块已经切到声明驱动，就更新统一声明，而不是再在多个文件里补散点逻辑。

### 6.4 导航接入要对

新增一级导航前，先判断：

- 是不是已有模块负责这个能力
- 是不是应该进现有页面的标签页而不是再开一级入口
- 是不是会制造新的平行入口

过去“自动发布功能”那类问题已经说明：

> **能收进已有页面的标签体系，就不要轻易再加一级全局入口。**

## 7. 登录态与 bootstrap 规则

Control UI 的登录态不是单点问题，它和入口链路直接绑在一起。

过去的经验已经证明：

- query 参数在多段跳转里更容易丢
- hash 作为启动桥更稳
- 只在 `launch.html` 做一次 bootstrap 不够稳
- `index.html` 在 bundle 加载前做二次 bootstrap 更稳

但结合今天的现行基线，当前可复用的结论应写成：

- 启动器直接打开 `index.html#token=...&gatewayUrl=...`
- 如果仍保留 `launch.html`，它只能算兼容和兜底页
- token 相关排障必须同时检查：
  - 最终 URL
  - `gatewayUrl`
  - `sessionStorage`
  - `localStorage`
  - `window.name`

## 8. 验证必须怎么做

以后凡是改了导航、入口、路由、bootstrap、共享资源，都不能只做“源码看起来对了”的验证。

最低验证清单应包括：

1. `/health` 返回正常
2. 目标 UI 页面返回 200
3. 浏览器里真实 DOM 的导航文本、active 状态正确
4. 点击导航后落到正式路由，不落到旧别名
5. 如果涉及登录态，确认 token 与 `gatewayUrl` 实际可用
6. 如果涉及共享导航，确认页面实际读取的是目标共享来源
7. 如果涉及缓存修复，确认版本刷新后浏览器不再读旧资源

只要这几项没过，就不能算真正修好。

## 9. 明确废弃和禁止的做法

下面这些信息已经过时或已经被验证会误导后续维护，应该明确视为废弃。

### 9.1 已废弃的信息

- 把 `extensions/public/control-ui-custom` 当成在线真源
- 把 `extensions/public/nav-bar.html` 当成长期唯一真源
- 认为导航统一等于“把多份 HTML 改成一样”
- 继续把 `launch.html` 当成当前主入口
- 继续把 `/auto.html`、`/automation.html`、`/publish.html`、`/experience.html` 当作正式入口
- 继续依赖根路径 `/` 兜底跳转到正确页面

### 9.2 明确禁止的做法

- 为同一功能保留多个都能跑的真实入口
- 修改多个导航副本而不确认谁才是实际 served 源
- 新增页面却不注册正式路由
- 只修源码，不看真实 DOM 和真实请求路径
- 在 Windows 日常启动链里继续依赖 `--force` 这类不稳参数
- 默认使用 `-ForceRestart` 打开 UI

## 10. 后续操作的统一原则

如果把前面所有规则再压缩一次，后面做 OpenClaw 前端和导航相关改动时，可以固定只记这几条：

1. 先识别唯一正式入口，再动代码。
2. 先识别唯一共享来源，再改导航。
3. 入口分流必须清楚，不同页面不要共用模糊入口。
4. 当前过渡运行态允许共享导航 HTML，但长期目标必须收敛到模块声明生成。
5. 新页面先归模块，再补路由，再决定要不要进全局导航。
6. 只看源码不算验证，必须看真实 DOM、真实路由和真实页面返回。
7. 发现双入口、双副本、双路由时，先收敛主链，再谈功能。

## 11. 一页执行版清单

以后你或 AI 再改这套前端，建议按下面这张短清单执行：

### 改导航前

- 谁是当前共享导航真源
- 这个入口属于哪个模块
- 这个入口是不是正式长期入口
- 会不会制造新的平行入口

### 改页面前

- 页面是不是放在模块 `frontend/pages`
- 路由是不是同步注册
- 是否还残留旧别名路由

### 改启动链前

- 当前唯一真实配置源是谁
- 当前浏览器主入口是不是 `index.html#...`
- 有没有把兼容页误当主入口

### 改完后

- `/health` 是否正常
- 页面是否 200
- DOM 是否正确
- 导航 active 是否正确
- token / gateway 是否正常
- 是否还残留旧入口、旧副本、旧脚本

## 结语

OpenClaw 的前端问题，从来不只是“一个导航栏”或者“一个页面入口”的问题。

真正会反复出错的，是这几件事总被拆开处理：

- 导航改一处
- 路由改一处
- 启动链改一处
- 配置源改一处
- 结果真实运行态还是另一套

所以以后这类工作只按一个原则做就够了：

> **把导航、入口、路由、配置源和验证当成一条链一起处理，优先收敛唯一正式路径，不再长期维护补丁式平行实现。**
