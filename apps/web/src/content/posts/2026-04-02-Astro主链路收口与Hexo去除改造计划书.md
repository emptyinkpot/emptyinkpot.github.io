# Astro主链路收口与Hexo去除改造计划书

日期：2026-04-02

## 1. 计划目标

本计划书用于指导当前仓库从“Astro 为主、Hexo 兼容保留”的过渡态，收敛为：

- **Astro 单一主链路**
- **Hexo 全量去除**
- **内容、构建、部署、文档事实源统一收口**
- **目录结构逐步对齐《通用文件与模块开发规范》**

本次改造不再保留 Hexo 兼容链路，不再保留 `legacy:*` 命令，不再把 `source/` 与 `themes/` 视为正式生产主路径。

## 2. 当前问题概述

当前仓库的主要问题不是 Astro 不可用，而是“事实源仍然分裂”：

- 根目录脚本已经把主入口切到 `apps/web/`
- 但根目录仍保留 Hexo 依赖和 `legacy:*` 命令
- 仓库仍保留 `source/`、`themes/`、Hexo 配置文件作为兼容层
- 部分文档仍混用“Hexo 历史真源”和“Astro 当前真源”的口径
- 顶层结构尚未完全对齐模块化规范，仍偏向迁移中间态

这会导致以下风险：

- 新开发容易误入旧链路
- 人与 AI 会对“真正入口”产生歧义
- 内容与路由事实源可能再次分叉
- 工程规则无法稳定收口到单一架构

## 3. 改造原则

本次改造遵守以下原则：

### 3.1 单一真源

- 页面真源收口到 `apps/web/`
- 内容真源收口到 Astro 当前使用的数据源
- 构建真源收口到 Astro 构建链路
- 部署真源收口到 Astro 产物

### 3.2 去兼容优先

- 不保留 Hexo 运行能力
- 不保留 Hexo 构建能力
- 不保留 Hexo 主题链路
- 不保留“先兼容、以后再删”的长期过渡口径

### 3.3 平台克制

仓库顶层只保留正式目录，不再让历史静态站脚手架继续以正式身份存在。

### 3.4 文档同步收口

任何仍声明 Hexo 为正式链路的说明，都要在本次改造中同步修正。

## 4. 目标架构

改造完成后，仓库目标形态为：

```text
project-root/
├─ apps/
│  └─ web/
├─ docs/
├─ tools/
├─ public-data/
├─ tests/
├─ .runtime/
├─ package.json
└─ README.md
```

说明：

- `apps/web/` 是唯一站点应用入口。
- `docs/` 是工程说明、规划与治理真源。
- `tools/` 是校验、生成、迁移脚本真源。
- `public-data/` 是当前已显式收口的公开数据真源目录之一，应继续保留并规范化使用。
- `source/`、`themes/`、Hexo 配置文件不再属于正式架构。

## 5. 本次改造范围

本次计划覆盖以下内容：

1. 删除 Hexo 依赖与脚本
2. 删除 Hexo 配置与主题目录
3. 明确 Astro 为唯一入口、唯一构建、唯一部署主链路
4. 统一根目录 README、执行文档、治理文档口径
5. 清点并迁移仍依赖 `source/` 的现存内容
6. 为后续进一步对齐模块化规范打基础

本次计划暂不强制一次性完成完整 DDD/模块化分层重构，但要完成“去旧链路、定主链路、定正式目录”的第一阶段收口。

## 6. 分阶段执行方案

## 阶段 A：事实源冻结

目标：

- 冻结 Astro 为唯一正式主链路
- 冻结 Hexo 为待删除历史资产

执行项：

1. 在根目录新增本计划书，作为本轮改造唯一总纲。
2. 在 `README.md` 中明确：
   - 唯一站点入口是 `apps/web/`
   - Hexo 已停止使用
   - 历史目录仅在删除前用于迁移核对
3. 梳理所有文档中涉及以下表述并统一修正：
   - “legacy”
   - “兼容”
   - “Hexo 仍可用于”
   - “source 仍为默认入口”

验收标准：

- 仓库文档不再把 Hexo 描述为有效运行链路。

## 阶段 B：Hexo 链路去除

目标：

- 从工程层面移除 Hexo 的运行、构建、主题、配置与依赖

执行项：

1. 删除根目录 `package.json` 中的以下脚本：
   - `legacy:clean`
   - `legacy:build`
   - `legacy:new`
   - `legacy:serve`
2. 删除根目录 Hexo 相关依赖。
3. 删除或归档后移除以下正式路径：
   - `_config.yml`
   - `_config.landscape.yml`
   - `themes/`
4. 评估并处理以下目录：
   - `source/`
   - `scaffolds/`
5. 若 `source/` 中仍有 Astro 主链路必须依赖的内容，则先迁移到 Astro 正式内容目录，再删除旧目录。

验收标准：

- 根仓库执行安装和构建时不再包含 Hexo。
- 仓库内不存在可被误认为正式入口的 Hexo 命令和主题目录。

## 阶段 C：内容真源收口

目标：

- 把内容生产、读取与公开展示统一收口到 Astro 使用的正式位置

执行项：

1. 全量盘点 `apps/web/` 当前读取哪些内容源。
2. 逐项确认以下内容是否仍依赖历史目录：
   - posts
   - notes
   - projects
   - pages
   - updates
3. 若 `apps/web/` 仍直接或间接读取 `source/`，则迁入 `apps/web/src/content/` 或其它正式 Astro 内容目录。
4. 更新内容治理脚本，确保它们校验的是新真源而不是历史兼容源。
5. 删除“兼容回退读取旧路径”的逻辑与文档口径。

验收标准：

- Astro 构建只依赖正式内容目录与正式公开数据目录。
- `source/` 不再承担任何生产职责。

## 阶段 D：入口与路由口径收口

目标：

- 让“入口、路由、初始界面、导航”全部基于 Astro 主链路定义

执行项：

1. 确认根命令仅保留：
   - `dev`
   - `build`
   - `preview`
   - `check`
2. 确认 Astro 首页 `/` 是唯一初始界面。
3. 确认导航、结构页、RSS、robots、sitemap 都由 `apps/web/src/pages/` 输出。
4. 复核 GitHub Actions 或部署脚本，确保部署产物只来自 Astro。

验收标准：

- 开发、构建、预览、部署都只走 Astro 主链路。

## 阶段 E：目录规范对齐

目标：

- 在不打断主站运行的前提下，把顶层目录向规范要求收敛

执行项：

1. 保留正式目录：
   - `apps/`
   - `docs/`
   - `tools/`
   - `public-data/`
2. 规划补齐以下正式能力目录：
   - `tests/`
   - `.runtime/`
3. 评估是否需要在后续阶段新增：
   - `kernel/`
   - `modules/`
4. 若当前站点仍是单应用内容站，可先不强行引入 `modules/`，但必须保证：
   - 正式目录清晰
   - 内容边界清晰
   - 公共逻辑不继续失控堆积

验收标准：

- 顶层目录仅保留正式职责目录。
- 历史目录不再与正式目录并列长期存在。

## 7. 关键改造清单

以下清单属于本轮必须完成项：

1. 删除 Hexo 脚本
2. 删除 Hexo 依赖
3. 删除 Hexo 配置文件
4. 删除 `themes/`
5. 迁移 `source/` 中仍被 Astro 使用的内容
6. 修正 README 与核心架构文档
7. 修正治理文档中的“兼容读取”表述
8. 确认部署只发布 Astro 产物

以下清单属于本轮建议完成项：

1. 为内容源建立正式目录说明文档
2. 为 `apps/web/` 建立更明确的边界说明
3. 补建顶层 `tests/`
4. 补建架构清单文档，例如 `ARCHITECTURE.md`

## 8. 风险与处理策略

### 风险 1：删除 Hexo 后丢失内容映射

处理策略：

- 删除前先完成内容盘点
- 以 Astro 实际读取链路为准逐项迁移
- 迁移完成后再删旧目录

### 风险 2：文档口径更新不全

处理策略：

- 对 `docs/`、`README.md`、维护日志进行关键词排查
- 把“兼容”“legacy”“旧入口”统一改成历史说明或删除

### 风险 3：工具脚本仍引用旧路径

处理策略：

- 逐个检查 `tools/` 下脚本
- 优先修复“读取旧源”的脚本
- 再删除旧目录

### 风险 4：一次性追求完全模块化导致改造过大

处理策略：

- 先完成“主链路唯一化”
- 再做“目录规范化”
- 最后再评估是否需要引入 `kernel/` 与 `modules/`

## 9. 验收标准

本轮改造完成后，必须同时满足以下条件：

1. 根仓库不再包含 Hexo 命令与依赖
2. `apps/web/` 是唯一正式站点入口
3. 构建与部署只走 Astro
4. `source/`、`themes/`、Hexo 配置文件不再属于正式架构
5. 文档不再宣称存在 Hexo 兼容主链路
6. 内容、页面、公开数据的事实源已经明确

## 10. 建议实施顺序

建议按以下顺序执行，不要跳步：

1. 冻结文档口径
2. 盘点 Astro 实际内容依赖
3. 迁移仍依赖 `source/` 的内容
4. 删除 Hexo 脚本与依赖
5. 删除 Hexo 配置与主题目录
6. 更新 README、治理文档、执行清单
7. 验证构建、预览、部署
8. 清理历史残留目录

## 11. 后续第二阶段建议

当第一阶段完成后，再进入第二阶段：

- 建立正式 `tests/`
- 明确 `apps/web/` 内部边界
- 控制公共逻辑向 `lib/` 无序扩张
- 评估是否需要引入 `kernel/` 与 `modules/`
- 建立可自动校验的目录与依赖边界规则

## 12. 最终结论

这次改造的核心不是“把 Hexo 文件删掉”这么简单，而是把仓库从迁移态收敛为正式态：

- **只有一个主入口**
- **只有一条构建链路**
- **只有一套正式事实源**
- **只有一套对外说明口径**

先完成这一步，后续再做更深的模块化治理，工程才不会继续在“历史兼容”和“现行主链路”之间来回摇摆。
