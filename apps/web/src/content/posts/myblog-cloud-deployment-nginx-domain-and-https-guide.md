这篇文章记录的是当前 `MyBlog` 项目第一次真正落到云服务器上的过程。

它不是泛泛而谈的“怎么部署一个 Astro 站点”，而是基于当前仓库和当前服务器现状，把已经做过的动作、当前口径和后续建议完整收口。

本文重点回答 5 个问题：

- 当前服务器上已经有什么项目
- `MyBlog` 是怎么和现有项目并列部署的
- 为什么这次选了 `Nginx`
- 现在其他电脑能不能访问这个博客
- 正式域名和 `HTTPS` 是怎么真正落地的

如果只看结果，这次部署似乎很简单：

- 服务器上装一个 `Nginx`
- 把 Astro 的 `dist/` 上传上去
- 配一个站点文件
- 然后通过公网 IP 打开

但真正有价值的部分其实不是“命令敲了什么”，而是这次明确了几条后续会反复复用的部署原则：

- 已有项目在运行时，新项目要优先做并列部署，而不是覆盖部署
- 静态站优先交给专业 Web 服务器，而不是长期靠临时端口顶着
- 文档必须回填真实服务器信息，否则下一次维护一定会靠猜
- 裸 IP 能用只能说明“第一阶段完成”，不能说明“正式上线完成”

## 一、当前服务器现状

当前项目对应的云服务器信息如下：

- 主机 IP：`124.220.233.126`
- SSH 用户：`ubuntu`
- SSH 命令：`ssh ubuntu@124.220.233.126`
- 服务器主机名：`VM-0-12-ubuntu`

服务器上已经存在一个现有项目：

- 项目名：`OpenClaw`
- 目录：`/srv/openclaw`
- 当前服务端口：`5000`
- 健康检查地址：`http://124.220.233.126:5000/health`

本次新部署的博客项目则使用：

- 项目名：`MyBlog`
- 部署目录：`/srv/myblog`
- 静态站点目录：`/srv/myblog/site`

这意味着两者不是覆盖关系，而是**同级项目**。

这一步其实很重要，因为很多服务器部署出问题，不是因为不会装服务，而是因为一开始就没有把“新项目和旧项目是什么关系”想清楚。

这次如果处理错了，最坏情况会出现：

- 新项目覆盖旧项目目录
- 新项目抢走旧项目端口
- 新项目改坏现有反向代理
- 后续根本不知道哪个目录对应哪个项目

所以真正第一步不是“马上部署”，而是先判断：

> `MyBlog` 应该作为 `/srv/openclaw` 的子目录存在，还是应该作为 `/srv/myblog` 的独立同级项目存在？

这次的结论是：

- 应该独立
- 应该同级
- 应该分别拥有自己的目录和服务边界

## 二、为什么这次没有继续用“裸端口静态服务”

如果只是临时把 Astro 静态站跑起来，其实可以直接开一个 Node 静态服务，例如：

- `5001` 跑博客
- `5000` 继续跑 `OpenClaw`

但这次最终没有这么做，而是安装了 `Nginx`。

原因很实际：

- 服务器上已经有一个正式运行的项目
- 后续很可能不只这两个项目
- 博客本身是静态站，更适合交给专业 Web 服务器托管

所以这次选择 `Nginx` 的核心原因不是“它更高级”，而是：

> 当服务器上开始出现多个长期维护项目时，应该有一个统一入口层，而不是让每个项目都各自暴露一个公开端口。

为了让这个判断更清晰，可以直接对比一下两种思路。

### 方案 1：继续增加公开端口

例如：

- `5000` 给 `OpenClaw`
- `5001` 给 `MyBlog`
- 以后再加项目就继续 `5002`、`5003`

这种方案的问题在于：

- 外部入口会越来越乱
- 用户不容易记住
- 运维时容易忘记哪个端口对应哪个项目
- 后续加 `HTTPS` 会更麻烦

### 方案 2：引入统一入口层

例如：

- `80/443` 由 `Nginx` 统一接收请求
- 静态项目直接由 `Nginx` 提供文件
- 需要动态服务的项目由 `Nginx` 反向代理到内部端口

这种方案的优点是：

- 结构更稳定
- 更容易扩项目
- 更容易做域名和 HTTPS
- 更适合作为长期服务器架构

所以这次虽然只是部署一个博客，但本质上是在把服务器从“临时跑服务”往“正式站点托管”推进一步。

## 三、Nginx 是什么，为什么这次更适合

`Nginx` 可以把它理解成“网站入口层”。

它主要做三件事：

1. 接收外部请求
2. 提供静态文件
3. 把请求转发给后端服务

对当前服务器来说，它的优势很明确：

- 可以直接托管 `MyBlog` 这种 Astro 静态产物
- 不影响 `OpenClaw` 继续占用 `5000`
- 以后方便按域名拆分不同项目
- 后续接入 `HTTPS` 会更顺手

所以这次最终形成的结构是：

```text
MyBlog
- Nginx
- 80 端口
- /srv/myblog/site

OpenClaw
- 自己的服务进程
- 5000 端口
- /srv/openclaw
```

还可以再进一步理解成：

- `Nginx` 负责“门面”
- `MyBlog` 负责“静态内容”
- `OpenClaw` 负责“自己的后端服务”

这三者的职责已经不再混在一起。

## 四、这次实际做了哪些部署动作

### 1. 先确认服务器现状

本次先检查了服务器上已有内容，包括：

- `/srv/openclaw`
- 当前监听端口
- `openclaw-gateway.service` 状态

确认结果：

- `OpenClaw` 正在运行
- `5000` 端口正常监听
- 当前机器上没有安装 `Nginx`

### 2. 安装 Nginx

Ubuntu 上实际执行的是：

```bash
sudo apt-get update
sudo apt-get install -y nginx
```

安装后确认：

- `nginx.service` 已启用
- 服务状态正常

### 3. 处理代码部署方式

原本计划是：

- 服务器本地 `git clone`
- 服务器本地构建

但在实际操作时，服务器上的 Git 拉取链路出现了卡顿。

因此本次改用更稳的方式：

> 本机构建，再把 Astro 静态产物上传到服务器。

这对于 Astro 静态站来说是完全合理的，因为最终真正对外提供服务的本来就是 `dist/` 目录。

这里顺手沉淀一个经验：

> 对于静态站，部署方式不一定非要“服务器 clone 仓库再构建”，只要你能稳定拿到正确的构建产物，把 `dist/` 同步到服务器同样是标准做法。

这次改成“本机构建 + 上传产物”，还有几个实际好处：

- 避开服务器上 Git 拉取链路卡顿
- 避开服务器上重复安装前端依赖
- 本地先完成 `lint/check/build`，发布前更可控
- 服务器侧只负责托管静态文件，职责更单纯

### 4. 本地构建博客

本地执行：

```bash
npm run build
```

构建成功后，静态产物位于：

```text
apps/web/dist/
```

### 5. 上传静态产物到服务器

服务器上的目标目录是：

```text
/srv/myblog/site
```

上传后，站点文件已经位于服务器：

- `/srv/myblog/site/index.html`
- `/srv/myblog/site/_astro/*`
- `/srv/myblog/site/pagefind/*`

### 6. 配置 Nginx 指向 MyBlog

当前 Nginx 配置文件：

```text
/etc/nginx/sites-available/myblog.conf
```

当前配置口径是：

```nginx
server {
    listen 80 default_server;
    listen [::]:80 default_server;
    server_name _;

    root /srv/myblog/site;
    index index.html;

    location / {
        try_files $uri $uri/ /index.html;
    }
}
```

它的作用是：

- 让服务器 80 端口默认返回 `MyBlog`
- 对 Astro 路由走 `index.html` 回退

### 7. 验证 MyBlog 和 OpenClaw 是否都正常

最终验证结果如下：

- `http://124.220.233.126/` 返回 `200`
- 返回内容已经确认是 Astro 生成的 `MyBlog` 首页
- `http://124.220.233.126:5000/health` 返回 `200`

这说明：

- 博客已经正常上线
- 现有 `OpenClaw` 没有被破坏

这一步不要只看“首页能不能打开”，而要看两件事是不是同时成立：

1. 新项目已经可访问
2. 老项目仍然正常

如果只验证第一条，不验证第二条，很容易出现一种假成功：

- 博客看起来上线了
- 但旧项目被你改坏了

而这次部署的目标明确不是替换 `OpenClaw`，而是和它并存。

## 五、现在其他电脑能不能访问这个博客

结论是：

> 正常情况下可以。

本次已经确认过这些前提：

- 服务器是公网 IP：`124.220.233.126`
- `Nginx` 已监听 `80` 端口
- 服务器本机访问 `http://127.0.0.1/` 返回 `200`
- 本机访问 `http://124.220.233.126/` 也返回 `200`
- 服务器防火墙当前未拦截：
  - `ufw` 状态为 `inactive`
  - `iptables` 默认策略为 `ACCEPT`

因此，只要云厂商控制台安全组没有额外拦截 `80` 端口，其他电脑就应该能直接访问：

```text
http://124.220.233.126/
```

这里要特别区分三层判断：

### 第 1 层：服务本身有没有起来

这次已经确认：

- `nginx` 在监听 `80`
- `openclaw-gateway` 在监听 `5000`

### 第 2 层：服务器本机有没有拦

这次已经确认：

- `ufw` 未启用
- `iptables` 默认策略为 `ACCEPT`

### 第 3 层：云平台安全组有没有拦

这一层不在服务器里，而是在云厂商控制台里。

也就是说，如果未来出现这种情况：

- 服务器本机一切正常
- 你本机也偶尔能访问
- 但别人访问失败

就应该优先去看：

- 云服务器控制台的安全组
- 是否明确放行 TCP `80`
- 如果要上 HTTPS，还要放行 TCP `443`

所以这次关于“别人能不能访问”的最准确结论应该写成：

> 服务器和服务本身已经满足公网访问条件，外部访问是否稳定还取决于云平台安全组是否放行 `80/443`。

## 六、现在为什么还不算“正式完成”

虽然博客已经能通过 IP 打开，但它还只是**可访问状态**，还不算理想的正式对外状态。

原因主要有三点：

### 1. 现在还是裸 IP 访问

这意味着：

- 不够好记
- 不够正式
- 后续如果服务器 IP 变化，访问地址也会跟着变

### 2. 现在还是 HTTP

这意味着：

- 浏览器不会把它当成完整可信站点
- 后续如果加表单、评论、登录、API 等功能，体验和安全感都不够好

### 3. 还没有形成“正式外网入口”

现在的入口是：

- `http://124.220.233.126/`

更适合长期维护的入口应该是：

- `https://blog.your-domain.com`

还有一个经常被忽略的问题是：

- 现在虽然已经部署了
- 但还没有形成“后续谁来更新、怎么更新、更新完怎么验证”的稳定运维动作

如果这一步不补上，第一次部署成功不代表以后每次更新都能顺。

## 七、为什么要继续绑定正式域名

给博客绑定正式域名，最大的好处不是“看起来更高级”，而是它会直接改变后续维护方式。

好处包括：

- 更像正式网站
- 更容易记忆和分享
- 以后如果服务器 IP 变了，只需要改 DNS，不需要通知所有人换地址
- 更方便和其他项目按域名分流

例如以后你完全可以这样规划：

```text
https://blog.example.com        -> MyBlog
https://openclaw.example.com    -> OpenClaw
```

这样就不需要长期靠 `:5000` 这种端口来区分项目。

再说得更直接一点，域名真正带来的不是“好看”，而是：

- 入口稳定
- 资产可迁移
- 服务边界更清楚
- 运维成本更低

哪怕以后你把博客迁到另一台服务器，只要域名不变，外部世界感知到的入口也不需要变化。

## 八、为什么要加 HTTPS

这一步很重要，原因主要有 6 个：

### 1. 浏览器信任度更高

很多浏览器会把纯 `http` 页面标成“不安全”。

### 2. 传输过程更安全

用户访问网站时，浏览器和服务器之间的数据会被加密。

### 3. 更适合长期演进

哪怕今天只是静态博客，后面如果加：

- 评论
- 表单
- 登录
- API

`HTTPS` 都会更像基础设施，而不是可选项。

### 4. SEO 和分享体验通常更好

搜索引擎、浏览器、部分平台对 HTTPS 站点通常更友好。

### 5. 更符合正式站点心智

一个正式博客更自然的访问方式应该是：

```text
https://your-domain.com
```

而不是：

```text
http://123.123.123.123
```

### 6. 后续运维更清晰

域名 + HTTPS 一旦配置好，博客的正式入口就会稳定下来，README 和部署文档也更容易长期维护。

从实际使用体验来说，`HTTPS` 还有一个非常现实的价值：

- 当你把博客地址发给别人时，对方看到的是一个正常可信的网站入口
- 而不是浏览器先跳一个“不安全”的提示或警告心智

这对“个人博客是否像一个正式项目”影响很大。

## 九、后续该怎么做：域名 + HTTPS

推荐的下一步动作是：

### 第一步：准备一个域名

本次实际使用的是：

- `blog.tengokukk.com`

### 第二步：把域名解析到当前服务器

本次实际在域名服务商后台添加了一条 `A` 记录：

```text
主机记录：blog
记录类型：A
记录值：124.220.233.126
```

也就是把：

```text
blog.tengokukk.com -> 124.220.233.126
```

绑定到当前云服务器。

### 第三步：修改 Nginx 的 `server_name`

把：

```nginx
server_name _;
```

本次实际改成：

```nginx
server_name blog.tengokukk.com;
```

### 第四步：安装 certbot

```bash
sudo apt-get update
sudo apt-get install -y certbot python3-certbot-nginx
```

### 第五步：申请 Let’s Encrypt 证书

```bash
sudo certbot --nginx -d blog.tengokukk.com
```

本次实际执行命令时，还附带了邮箱和自动跳转参数：

```bash
sudo certbot --nginx -d blog.tengokukk.com --non-interactive --agree-tos -m 1915*****5@qq.com --redirect
```

证书签发成功后，`certbot` 自动把 `Nginx` 配置更新为：

- `80` 端口自动 `301` 跳转到 `HTTPS`
- `443` 端口挂载证书
- 证书路径位于 `/etc/letsencrypt/live/blog.tengokukk.com/`

### 第六步：放通云平台 `443` 端口

这一步非常关键，而且这次就是在这里补齐了最后一个阻塞点。

当时服务器内已经确认：

- `Nginx` 正在监听 `443`
- 服务器本机访问 `https://127.0.0.1/` 返回 `200`
- 证书也已经成功签发

但外网访问 `https://blog.tengokukk.com/` 仍然超时。

这说明问题已经不在服务器内部，而是在云平台入口层。最终排查发现，腾讯云轻量应用服务器控制台里还没有放行：

- `TCP 443`

补上这条防火墙规则之后，外网 `HTTPS` 立刻恢复正常。

这一步也顺手沉淀出一个很实用的判断顺序：

1. 先看服务是否在监听端口
2. 再看服务器本机是否能访问
3. 最后看云平台安全组或防火墙是否放行

如果前两层都正常，而外部仍然超时，优先怀疑云厂商控制台，而不是继续在服务器里反复改配置。

### 第七步：确认正式入口已经可用

本次最终已经验证通过：

- `http://blog.tengokukk.com/` 返回 `301`
- 自动跳转到 `https://blog.tengokukk.com/`
- `https://blog.tengokukk.com/` 返回 `200`

也就是说，博客的正式入口现在已经变成：

```text
https://blog.tengokukk.com/
```

这比裸 IP 入口：

```text
http://124.220.233.126/
```

更适合作为长期对外地址。

如果后续你希望博客和现有项目都更正式，还可以进一步规划成：

```text
https://blog.example.com         -> MyBlog
https://openclaw.example.com     -> OpenClaw
```

这样长期会比：

```text
http://124.220.233.126/
http://124.220.233.126:5000/
```

更像一个清晰的多项目服务器结构。

## 十、当前阶段的最终口径

当前 `MyBlog` 的真实部署状态是：

- 已经成功上线到云服务器
- 已经和 `OpenClaw` 以同级项目方式共存
- 已经通过 `Nginx` 在 `80/443` 端口提供服务
- 已经可以通过公网 IP 直接访问
- 已经绑定正式域名 `blog.tengokukk.com`
- 已经启用 `HTTPS`
- 已经完成 `HTTP -> HTTPS` 自动跳转

所以最准确的描述应该是：

> `MyBlog` 已经完成“正式域名 + HTTPS + 外网可访问”的当前阶段部署，下一阶段重点会转向持续更新、自动化发布和运维收口。

## 十一、给以后自己的运维提醒

如果后面你继续维护这台服务器，建议固定记住下面这条顺序：

1. 先确认当前项目边界
2. 再确认当前入口层是谁
3. 先做本地构建和检查
4. 再同步静态产物
5. 最后验证“新项目正常 + 旧项目未受影响”

对这台机器来说，当前最简化的运维口径就是：

- `OpenClaw`：继续走自己的服务链路
- `MyBlog`：继续走 `Astro static + Nginx`
- 正式入口：后续逐步从裸 IP 过渡到域名 + HTTPS
