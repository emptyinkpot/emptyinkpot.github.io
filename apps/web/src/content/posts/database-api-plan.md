这篇文章的目标不是讨论某个数据库驱动怎么调，而是先把一件事讲清楚：**如果希望客户端长期稳定访问数据能力，就不要让客户端绑定数据库文件路径，而要让客户端绑定稳定的服务地址与接口协议。**

## 先说结论

如果未来要让 Roo、脚本工具、桌面程序或者其他客户端都能稳定访问数据库，那么最可靠的方案不是让每个调用方直接摸数据库文件，也不是把数据库路径硬编码进客户端，而是单独提供一个数据库 API 服务。

客户端只需要知道：

- 服务地址是什么
- 接口协议是什么
- 请求和响应格式是什么
- 自己具备什么调用权限

这样后续无论数据库文件搬迁、项目目录调整、部署环境变化，还是服务内部重构，客户端侧都不需要跟着大改。

## 为什么不应该直接暴露数据库文件路径

### 1. 路径是最脆弱的耦合点

一旦客户端依赖某个固定路径，例如某个 SQLite 文件、某个工程目录下的相对位置，那么下面这些变化都会把系统拖垮：

- 仓库迁移
- 目录重构
- 运行环境从本地换到服务器
- 单机运行换成独立服务运行
- 数据库从本地文件换成远程实例

也就是说，路径只能描述“现在文件在哪”，不能描述“系统能力该怎么被稳定调用”。

### 2. 客户端不应该知道服务端内部结构

客户端真正关心的是能力，不是目录。

例如客户端需要的是：

- 查用户
- 查项目
- 写记录
- 拉取状态

它不应该知道：

- 服务端把数据库放在哪个磁盘
- 数据访问代码放在哪个目录
- 当前仓库是不是又拆分了一层 `src/` 或 `server/`

一旦客户端开始依赖这些内部结构，服务端就失去了演进空间。

### 3. 直接共享数据库文件会扩大风险面

如果多个调用方直接接触数据库文件，就会带来：

- 权限边界难控制
- 连接管理混乱
- 并发写入风险变高
- 日志与审计难统一
- 未来升级数据库类型时迁移成本极高

所以稳定做法应该是：**客户端访问服务，服务访问数据库。**

## 应该绑定什么：服务地址 + 协议

真正稳定的识别方式，不是文件路径，而是下面这组契约：

- 服务地址
- API 版本前缀
- 接口资源路径
- 请求参数结构
- 响应结构
- 错误模型
- 鉴权规则

只要这组契约稳定，服务端内部怎么换实现都可以被控制在边界内。

## 为什么 API 服务模型更稳定

把数据库访问收口为服务之后，会立刻得到几个长期收益。

### 1. 客户端与数据库解耦

客户端不再依赖数据库类型、驱动方式和文件位置。

### 2. 服务端可独立演进

服务端以后可以逐步做这些事情，而不需要把影响直接扩散到所有客户端：

- SQLite 换 PostgreSQL
- 本地运行换 Docker 或系统服务
- 单文件查询换 Repository 分层
- 平铺目录换模块化目录

### 3. 安全边界更清晰

鉴权、日志、限流、审计都可以统一落在 API 层，而不是让每个调用方自己各玩各的。

### 4. 配置可以外置

数据库连接串、服务端口、运行环境可以通过环境变量统一管理，而不是写死在多个地方。

### 5. 未来更容易版本化

一旦接口带上版本前缀，后续升级就有明确落点，不会因为改一处接口把所有客户端打崩。

## 最小可行方案是什么

如果先做一个能稳定上线的最小版本，我建议结构至少包括下面这些部分：

- 配置读取层
- 数据库连接层
- 鉴权中间件
- Repository 层
- Service 层
- Controller 层
- 路由层
- 服务入口

这个结构看起来比“一个文件写完”复杂一些，但它能避免后面越写越乱。

## 推荐的基础目录结构

最初可以先从相对清晰的技术分层开始：

```text
src/
  config/
  db/
  middleware/
  repositories/
  services/
  controllers/
  routes/
  utils/
  index.js
```

这套结构适合第一阶段快速把服务跑起来，尤其适合先做：

- 数据库连接统一收口
- 路由挂载
- 权限校验
- 错误处理
- 基础资源接口

## 为什么后面还要继续升级成模块化结构

如果接口越来越多，继续把所有控制器、服务、路由都堆在全局目录里，会出现几个典型问题：

- 同一个业务上下文被拆散到很多目录
- 找一个功能要跨很多位置跳转
- 新功能没有明确归属
- AI 更容易误改到不相关代码
- 模块边界越来越弱

所以第一阶段可以技术分层启动，但第二阶段最好收敛到 `src/modules/*` 的闭环结构。

## 模块化后的推荐方向

更长期稳定的结构应该更像这样：

```text
src/
  kernel/
  modules/
    users/
      contracts/
      application/
      domain/
      infrastructure/
      interface/
    projects/
      contracts/
      application/
      domain/
      infrastructure/
      interface/
  index.js
```

这意味着每个业务模块都尽量形成完整闭环，而不是把业务拆散到全局技术目录里。

## API 入口为什么应该统一到 `/api/v1`

统一版本前缀是稳定服务的关键纪律之一。

典型挂载方式例如：

```js
app.use("/api/v1", authMiddleware);
app.use("/api/v1/users", userRoutes);
app.use("/api/v1/projects", projectRoutes);
```

这样做有几个直接好处：

- 未来可以并行存在 `v1` 和 `v2`
- 鉴权中间件可以统一挂在版本前缀下
- 日志、错误处理、监控更容易统一接入
- 客户端只依赖稳定前缀，不依赖零散裸路由

## 为什么不能开放任意 SQL 执行接口

从表面上看，开放一个“传 SQL 过来我替你执行”的接口似乎最灵活，但这其实是最不稳定也最危险的方案。

问题包括：

- 安全风险过高
- 权限边界模糊
- 客户端被迫理解数据库实现细节
- 接口契约无法稳定
- 后续重构数据库结构时几乎无法收口

正确方向应该是按资源和业务能力暴露接口，而不是把数据库变成远程命令行。

## Repository、Service、Controller 为什么要分层

这三层不是为了“看起来专业”，而是为了让职责分工稳定下来。

### Repository

负责数据库访问收口。

- SQL 放这里
- 数据查询和持久化细节放这里
- 不让控制器或业务层到处直接碰数据库

### Service

负责业务逻辑与业务编排。

- 组合多个查询
- 校验业务规则
- 处理事务边界
- 聚合领域行为

### Controller

负责 HTTP 入口与协议适配。

- 接收请求
- 调参数校验
- 调 Service
- 返回统一响应

只要这三层守住，后面加日志、统一错误、测试、审计都会轻松很多。

## 参数校验、统一响应、统一错误模型要尽早定下来

如果只是先把接口做出来，后面再慢慢补统一规则，通常结果就是永远补不齐。

所以建议一开始就确定：

### 1. 统一响应格式

例如所有接口都返回稳定的成功/失败结构，让客户端不需要对每个接口写不同适配。

### 2. 统一错误模型

错误要可分类、可追踪，而不是每个地方随手抛一个字符串。

### 3. 统一参数校验

建议把参数校验前移，最好结构化，避免脏数据进入业务层或数据库层。

后续如果引入 Zod 或类似工具，整个契约会更稳。

## 配置应该怎么放

配置不要散落在代码里，也不要依赖大量相对路径猜测。

建议至少把这些内容放到环境变量或集中配置入口：

- 服务端口
- 数据库连接信息
- 鉴权密钥
- 日志级别
- 运行环境标识

这样做的意义是：

- 部署切换更容易
- 敏感信息不必硬编码
- 客户端与服务端职责更清楚
- 服务能更自然地做成系统服务或常驻进程

## 为什么要考虑做成系统服务或常驻进程

如果数据库 API 是长期能力，而不是一次性脚本，那么它最好具备持续运行、自动拉起、异常重启和日志可观测能力。

这也是后面引入 PM2 一类进程管理器的原因。

一个典型配置示意如下：

```js
module.exports = {
  apps: [
    {
      name: "data-api-server",
      script: "./src/index.js",
      cwd: __dirname,
      instances: 1,
      exec_mode: "fork",
      autorestart: true,
    }
  ]
};
```

这样后续就更容易：

- 长期运行
- 统一看日志
- 管理进程状态
- 接入开机自启

## 模块命名为什么要做成硬规则

如果未来要升级到模块化结构，模块命名必须提前收紧，否则目录、路由、声明文件和生成器都会变得很混乱。

例如可以强制模块名使用 kebab-case：

```js
function validateModuleName(name) {
  if (!name) {
    throw new Error("Module name is required.");
  }

  const kebabCasePattern = /^[a-z0-9]+(?:-[a-z0-9]+)*$/;

  if (!kebabCasePattern.test(name)) {
    throw new Error(
      "Invalid module name. Use kebab-case only, e.g. users, projects, novel-chapters."
    );
  }
}
```

这类规则看起来很小，但它会直接影响：

- 模块目录稳定性
- 自动生成器是否可用
- 路由与文档命名是否一致
- AI 自动生成内容是否容易跑偏

## 为什么自动模块生成器是下一步关键升级

当规范写到一定程度，如果还完全靠人工手动复制目录和文件模板，执行成本会越来越高。

所以后续应该补上自动模块生成器，把下面这些动作程序化：

- 创建模块骨架
- 生成标准目录
- 生成 `module.json` 或其他声明文件
- 注入基础入口
- 建立测试目录
- 做命名校验

这样规范才不只是文档，而是可执行能力。

## 迁移顺序应该怎么走

如果现在还是一个平铺服务，不建议一下子全量重做。更合理的顺序是：

### 第一阶段：建立最小稳定 API 服务

先把：

- 服务入口
- 配置入口
- 数据访问收口
- 基础路由
- 鉴权
- 错误处理

做出来。

### 第二阶段：统一版本前缀与协议层纪律

再收敛：

- `/api/v1`
- 统一响应
- 统一错误模型
- 参数校验

### 第三阶段：模块化收口

把按技术类型平铺的结构逐步迁移到 `src/modules/*`。

### 第四阶段：工具化与门禁化

补上：

- 模块生成器
- 命名校验
- 结构校验
- 测试门禁
- 质量检查

## Roo 这类客户端应该怎么接入

如果未来由 Roo 或其他 AI/自动化工具访问数据库能力，建议遵守两个原则：

### 1. Roo 只连 API，不直连数据库

这样客户端职责最单纯，也最不容易把数据库账号、路径、内部结构暴露出去。

### 2. 敏感配置尽量留在服务端

数据库账号、连接串等应优先留在服务端环境，而不是散落在客户端本地配置中。

## 这套方案真正解决的是什么问题

表面上看是在做一个数据库接口服务，实际上是在解决更长期的几个问题：

- 如何避免路径耦合
- 如何降低目录重构对客户端的冲击
- 如何给系统建立稳定边界
- 如何为未来模块化和自动化留出生长空间
- 如何让 AI、脚本和人工开发都共享同一事实源

## 最终建议

如果只用一句话概括这份计划，那就是：

**客户端不要绑定数据库路径，而要绑定稳定的 API 服务；服务端不要堆成平铺脚本，而要朝模块闭环、协议稳定、配置外置、可校验、可生成的工程结构演进。**

## 下一步执行清单

落地时建议按下面这个顺序推进：

1. 先搭一个最小数据库 API 服务骨架
2. 统一挂到 `/api/v1`
3. 收敛配置与鉴权入口
4. 让 Repository、Service、Controller 职责分离
5. 为后续模块化预留 `src/modules/*` 方向
6. 补统一响应、统一错误和参数校验
7. 再推进模块生成器与结构校验工具

这样做，后面无论数据库、目录、部署环境还是调用方发生变化，系统都更容易稳住。
