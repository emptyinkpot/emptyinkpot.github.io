## 一、先说总结论：这类问题的核心从来不是“怎么改一个 key”，而是“谁才是当前活动真源”

这组原文表面上分别在讨论：

- Roo 改 key
- Codex 改 key
- VS Code 运行态配置
- 本地 MCP / skill / retrieval 结构
- Roo 扩展故障修复

但它们最终都指向同一个核心判断：

**本机 VS Code 里的 Roo / Codex 问题，绝大多数都不是“不会写配置”，而是没有先确认当前运行态到底在读哪份配置。**

如果活动真源没搞清楚，就会出现这些典型错觉：

- 明明改了文件，但运行的还是旧 key
- 明明目录里有配置，但扩展根本没读它
- 明明把 `.env.local` 改了，但 Roo 仍然不生效
- 明明磁盘上 MCP 都在，但当前运行态没有接入
- 明明修了存储文件，真正的故障却在扩展 bundle

所以这套总稿首先要统一的，就是“活动真源”这条主线。

## 二、先认清 VS Code 里的三层事实

根据原有运维与修复文，可以把问题来源拆成三层：

### 第 1 层：VS Code 当前运行态真源

这是最优先的一层。因为你眼前的扩展行为，首先是由 VS Code 当前会话加载到内存中的活动配置决定的。

### 第 2 层：扩展私有持久化目录

Roo 扩展自己的存储、设置、MCP 配置、历史状态，往往都落在 VS Code 用户目录下的私有存储区域中。

### 第 3 层：磁盘上的候选文件和历史文件

例如手工备份、旧配置副本、`.env.local`、历史目录、旧存储快照。这些文件可能有参考价值，但不一定是当前运行态真源。

这三层的优先级一定要分清：

**先看运行态，再看当前有效持久化，再看历史候选文件。**

## 三、本机关键位置为什么必须先记住

原文里反复强调了“本机关键位置总表”，这是因为此类问题不能靠猜路径解决。

至少要建立下面这类位置意识：

- VS Code 用户配置目录
- Roo 扩展安装目录
- Roo 扩展私有存储目录
- 当前 Roo MCP 配置文件位置
- Codex 的配置与鉴权文件位置
- 本地技能目录和本地能力仓位置

如果你连这些位置都没有统一地图，之后每次排查都会陷入“好像哪里都有一份配置”的混乱。

## 四、Codex 改 Key 的标准做法

原运维指南已经给出了很明确的做法，这里统一收编成固定流程。

### 1. 先备份两个关键文件

在正式修改前，先备份当前鉴权和配置文件，避免因为误改导致完全失去对比基线。

### 2. 改 `auth.json`

Codex 的认证信息首先要在 `auth.json` 中调整。这个文件通常承载最核心的鉴权信息。

### 3. 改 `config.toml`

如果涉及 base URL、模型、供应商或相关行为配置，还需要同步检查并调整 `config.toml`。

### 4. 不要误改 `.codex-global-state.json`

原文特别提醒这一点，说明它很容易被误判成“主配置文件”，但它并不是标准改 key 的主要入口。

### 5. 改完后一定要验证

修改后必须验证当前 VS Code / Codex 会话是否实际切到新配置，而不是只停留在文件层面“看起来改了”。

## 五、Roo 改 Key 的标准做法

Roo 的情况之所以看上去更麻烦，不是因为它设计得一定更复杂，而是因为很多人直接改到了非活动真源。

正确顺序应该是：

### 第 1 层：先看 VS Code 运行态真源

先确认当前 Roo 运行会话到底认的是哪份活动配置，以及当前扩展是否还在缓存旧状态。

### 第 2 层：再检查 Roo 扩展私有存储

Roo 的关键配置和 MCP 投影，往往不在你直觉以为的仓库根目录，而在扩展私有持久化目录。

### 第 3 层：把 `.env.local` 降级为历史候选文件

原文已经明确指出：`.env.local` 不应再被视为唯一或最高优先级事实源，它更多只是历史候选文件或辅助配置来源。

### 第 4 层：最后刷新运行态

改完之后如果不刷新 VS Code 扩展宿主或不 reload，很多情况下你看到的仍然是旧运行态。

## 六、为什么“明明改了，为什么还是旧 key”是 Roo 最常见故障

这类问题本质上有几类高频原因：

### 1. 改错位置

修改的是历史副本、候选文件、备份文件，而不是当前活动配置。

### 2. 运行态没刷新

即使磁盘已经改对了，但扩展会话仍在使用旧缓存。

### 3. Base URL 或配套项没有同步

只改 key 不改 base URL，或者反过来只改 URL 不改鉴权，也会形成“像是 key 错了”的假象。

### 4. 私有存储投影没有对齐

当前运行态真正读取的是扩展私有存储中的投影，而不是你刚改的另一个来源。

## 七、Base URL 为什么也是一个高频坑

原运维文专门提醒了 Base URL，这一点必须单独保留。

原因是很多鉴权问题表面像 key 错误，实际却是：

- 请求发到了错误的服务地址
- 使用了不匹配的供应商接口
- 配置项之间组合不一致

所以排查时不能只问“key 对不对”，还要问：

- 当前请求发到哪里
- 当前模型/供应商是谁
- 当前 base URL 和 key 是否属于同一套服务

## 八、Roo 的 MCP / Skill 配置真正该看哪里

这是这组文档里非常关键的一条事实：

**Roo 的 MCP / Skill 问题，不能只看仓库里的某份示例文件，而要看当前扩展活动配置真正指向的入口。**

原文强调的重点可以归纳为：

- 真正该看的入口必须先确认
- 改 MCP 时要知道哪些地方需要同步
- 改完后一定要 reload

也就是说，MCP 配置不是改完磁盘文本就自动生效，而是一个“配置文件 + 活动运行态 + 扩展刷新”共同成立的结果。

## 九、本机 Roo MCP 当前结构的关键认识

根据本机检索索引文，可以把 Roo 侧本地能力结构理解为几层：

### 1. Roo Code 扩展安装目录

这里承载扩展本体，但不代表所有可变运行配置都应该直接在这里改。

### 2. Roo Code 的 VS Code 持久化目录

这里更接近当前用户级活动配置、存储和状态的实际落点。

### 3. Roo 本地能力仓

这里承载本地 skill、MCP 相关能力、探针或周边能力资源。

### 4. 活动 MCP 配置

真正决定当前可接入 MCP 的，是当前活动 `mcp_settings.json` 一类配置，而不是“磁盘上某个 server 存在就等于已启用”。

## 十、当前机器上的 MCP 现状说明了什么

原索引文已经把现状说得很清楚：

- `experience-manager` 已接入 Roo MCP 配置
- 当前 Roo 中的 `experience-manager` 更像包装器/探针
- Roo 现有 skill 会主动调用 `experience-manager`
- 当前没有直接发现 `.roo` 侧的 `qmd` skill
- `custom_modes.yaml` 当前为空

这些判断说明一个关键点：

**本机上的“能力存在于磁盘”与“能力已经进入当前运行链”不是一回事。**

所以以后排查 MCP 时，必须分开看：

- 服务是否存在于磁盘
- 是否出现在活动 MCP 配置中
- 当前 skill / 模式是否真的会调它
- 当前运行态是否已经 reload 并接入它

## 十一、Codex 本地技能与本地检索结构怎么理解

原索引文对 Codex 一侧也给出了几条很关键的结论。

### 1. Codex 本地技能目录是独立事实源之一

Codex 的 skill 目录应被视为与 Roo 不同的一条能力接入链。

### 2. `qmd` skill 已明确写了“先查再答”

这说明 Codex 侧本地检索不是一个模糊概念，而是有明确 skill 行为约束的。

### 3. `qmd` 指向了经验镜像路径

这说明检索能力依赖的不只是模型本身，还依赖本地知识镜像和路径映射。

### 4. 经验镜像需要有刷新入口

如果镜像不刷新，再强的检索策略也会建立在过时知识上。

## 十二、Roo 与 Codex 在本机的功能分工

把几篇文章合起来，可以得出一个更清晰的功能分工图：

- Roo：更偏向 VS Code 扩展运行态、模式、MCP 接入、私有存储、活动配置投影
- Codex：更偏向独立配置文件、技能目录、本地知识检索、技能驱动行为
- VS Code：承担扩展宿主、全局用户配置、窗口 reload、运行会话承载

这意味着排查时不能把三者混成一个黑盒，而应明确：

- 是扩展运行态问题
- 是本地 skill 问题
- 是 MCP 接入问题
- 还是鉴权与 base URL 的配置问题

## 十三、Roo 故障修复案例里最重要的经验是什么

修复记录文提供了一个非常典型的反例：

表面症状可能像是 Roo history/storage 损坏，但真正阻断启动的原因其实是**扩展 bundle 损坏**。

这条经验非常关键，因为它说明：

### 1. 不要只根据表层症状下结论

存储异常、历史损坏、配置不生效，有时只是表层现象。

### 2. 先做备份再修

原事故处理顺序里，第一步就是先创建备份，这是必须保留的纪律。

### 3. Roo storage 可以修，但不代表根因就在 storage

修存储是必要动作，但不能因此忽略扩展本体、bundle、宿主环境等更底层原因。

### 4. 修完后要把 MCP 链接恢复回活动干净存储

说明修复不是“能启动就算完”，而是还要把原有能力接回当前有效运行态。

## 十四、下次遇到 Roo / VS Code 故障，推荐排查顺序是什么

整合运维文、检索文和修复记录后，最稳的排查顺序可以固定为下面这套：

### 1. 先确认当前活动真源

先判断当前 VS Code 会话真正读取哪份配置、哪份私有存储。

### 2. 再检查鉴权与 Base URL 是否成套

确认 key、base URL、供应商和模型是否属于同一套配置组合。

### 3. 再看 MCP 活动配置是否正确

确认活动 `mcp_settings.json` 是否真的包含目标 MCP。

### 4. 再看 skill / mode 是否真的调用了目标能力

磁盘存在不代表被调用，配置存在也不代表进入主流程。

### 5. 必要时检查扩展私有存储和历史状态

用于排查配置投影失真、状态污染、旧缓存残留等问题。

### 6. 再考虑扩展 bundle 是否损坏

当配置、存储、MCP 都看起来正常却仍然无法工作时，必须把扩展 bundle 损坏纳入排查范围。

### 7. 最后做受控 reload 或扩展宿主重启

确保当前运行态真正刷新，而不是继续沿用旧内存状态。

## 十五、三套最常用的修复剧本

原运维文已经将常见修复方式总结成三套剧本，这里统一保留它们的思路：

### 剧本一：重置 Roo 当前活动配置名

适用于当前会话读取错了配置投影、活动配置指向混乱的场景。

### 剧本二：重建标准配置投影

适用于私有存储与标准配置之间已经脱节，需要把有效配置重新投影到活动运行态的场景。

### 剧本三：受控重启 VS Code 扩展宿主

适用于配置已对齐但运行态未刷新，或者扩展宿主状态异常的场景。

这三套剧本共同体现了一个原则：

**配置问题很多时候不是“多写几行文件”能解决，而是要把活动配置、投影关系和宿主刷新一起处理。**

## 十六、以后如果要继续改，本机优先应该改哪里

根据本地检索索引文，后续继续演进时，优先级大致应是：

### 第一优先：Roo 的模式与提示词入口

因为这决定了运行时行为如何被组织和触发。

### 第二优先：在 `.roo\skills` 中补一个通用检索优先 skill

这样才能把“先查再答”从经验做法升级成稳定行为约束。

### 第三优先：保持 `experience-manager` 与 `qmd` 镜像同步

否则 Roo 和 Codex 会建立在不同知识镜像上，导致行为逐步漂移。

## 十七、这几篇文章合并后，真正形成的长期操作原则

把 Key、MCP、检索、修复、VS Code 运行态全部合起来，最终沉淀出的长期原则主要有这些：

### 1. 先认清活动真源，再改配置

### 2. 鉴权必须和 Base URL 成套检查

### 3. MCP 是否生效，要看活动配置与运行态，不只看磁盘文件

### 4. Skill 是否存在，不等于当前流程一定会调用

### 5. `.env.local`、历史副本、备份文件都只能当候选参考，不应默认当真源

### 6. 任何修复前先备份，任何修复后都要 reload 并验证

### 7. 当表面症状解释不通时，要把扩展 bundle 损坏纳入真实可能性

## 十八、最小验证清单

为了避免“改了但没真生效”，这里把多篇文章中的验证动作收束成一个最小清单：

- 确认当前配置文件已按预期修改
- 确认当前活动真源就是被修改的那份
- 确认 key 与 base URL 匹配
- 确认活动 MCP 配置包含目标能力
- 确认对应 skill / mode 会调用目标能力
- 进行 VS Code reload 或受控扩展宿主刷新
- 实际发起一次功能验证，确认不是停留在静态文件层面的“看起来正确”

## 二十、一句话收束

如果把 Roo、Codex、VS Code、本地 MCP 和检索体系全部压成一句话，那就是：

**先确认当前运行态真正读哪份配置，再去改 key、改 MCP、改 skill、做 reload 和做修复，本机这类问题才会从“玄学”变成可控运维。**
