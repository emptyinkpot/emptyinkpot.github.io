这篇文章先作为草稿落在真正的 `myblog-source` 源码仓里，目的不是讲一个“通用静态站恢复教程”，而是把这次 `MyBlog` 的真实恢复过程收口成一份后续可继续扩写的主文。

这次恢复里，最关键的认知只有一句话：

> 线上静态站点是部署结果，不是可持续维护的编辑真源。

如果手里只有已经部署好的 `HTML/CSS/JS` 载荷，你当然还能把站点继续挂着，但你已经失去了下面这些真正重要的东西：

- 文章源文件和 frontmatter
- 分类、标签、系列这些内容治理结构
- 构建脚本与校验脚本
- 可重复的 lint / check / build 流程
- 后续继续写新文章的稳定入口

所以这次工作的重点，不是“把线上站点复制回来”这么简单，而是要恢复出一套能继续写、继续改、继续发布的源码仓。

## 一、这次恢复到底在比对什么

当前一共涉及 3 份不同性质的材料：

- 真正可编辑的源码仓：`E:\My Project\Atramenti-Console\codex\apps\myblog-source`
- 本地保留的一份静态站点载荷：`E:\My Project\Atramenti-Console\codex\plugins\obsidian\data\sites\MyBlog`
- 服务器当前实际部署目录：`/srv/myblog/site`

它们看起来都像“博客内容”，但语义完全不同：

- `myblog-source` 负责编辑、治理、构建和发布
- 本地静态载荷更像一次历史快照或备份
- 服务器目录只代表当前线上实际正在提供访问的结果

如果不先把这 3 层分清楚，后面就很容易陷入一个常见误区：

- 一会儿改本地静态文件
- 一会儿改线上目录
- 一会儿再猜源码应该长什么样

最后三个地方都像是真的，但没有一个地方能作为唯一真源。

## 二、恢复后的源码仓现在恢复到了什么程度

本轮先做的不是“盲写新文章”，而是先把源码仓恢复到至少可验证的状态。

在 `myblog-source` 里执行的基线验证包括：

```bash
npm run lint
npm run check
npm run build
```

本轮修复后，三项都已经通过。

这一步的意义非常直接：

- `lint` 通过，说明当前源码没有明显的前端静态错误
- `check` 通过，说明更新日志和内容治理规则没有坏
- `build` 通过，说明这套源码仍然能稳定产出 Astro 静态站

只有走到这一步，才谈得上“从线上结果重新找回了一个可以继续维护的源码仓”。

## 三、反向比对后，线上站点和恢复后的源码差了多少

这次对比不是只看目录名，而是按文件清单和哈希去看。

截至 2026-04-21，本轮比对得到的结论是：

### 1. 本地静态快照 vs 服务器线上目录

- 本地快照文件数：`234`
- 服务器线上文件数：`243`
- 同路径同哈希：`205`
- 同路径不同哈希：`0`

这说明一个非常重要的事实：

> 服务器当前线上站点，和本地保留的那份静态载荷总体上是同一代产物。

两者差异主要不在页面内容本身，而在：

- 一部分文件数量不一致
- 一部分分类 / 标签路径出现了中文文件名编码漂移
- 若干搜索索引碎片文件不同

也就是说，**静态快照层基本是对得上的**。

### 2. 恢复后的源码构建产物 vs 服务器线上目录

- 源码新构建产物文件数：`259`
- 服务器线上文件数：`243`
- 同路径同哈希：`15`
- 同路径不同哈希：`114`
- 仅新构建产物存在：`130`
- 仅服务器线上存在：`114`

这说明另一件更关键的事：

> 恢复后的源码仓已经不是“简单复刻当前线上站点”，而是已经演化成一套新的内容与结构主线。

这类差异大概率来自几个方向：

- 首页与导航结构已经被重新整理
- 分类、标签、系列页的命名口径变了
- 资源文件名和打包产物哈希变了
- Pagefind 搜索索引是按新内容重新生成的

所以当前状态不是“源码仓恢复失败”，而是：

- 线上仍在跑旧一代静态产物
- 恢复出的源码仓已经进入新的可维护主线

## 四、为什么这反而是个好消息

表面上看，源码构建结果和线上差异很大，好像不太安心。

但从维护角度看，这其实是好消息，因为它说明我们已经重新拿回了真正的编辑权。

如果比对结果是“源码仓只能勉强拼出一份和线上完全一样的残缺载荷”，那意味着我们并没有恢复维护能力，只是重新打包了一次历史快照。

而现在的状态是：

- 真源码仓能跑校验
- 真源码仓能成功构建
- 真源码仓里已经存在完整内容目录
- 真源码仓里可以直接继续新增和修改文章

这才是后续持续写博客真正需要的基础。

## 五、接下来真正要做的，不是再纠结哪份静态文件更像

到这一步之后，优先级应该切换。

下一阶段最重要的事情不再是继续把所有静态文件逐个对齐，而是：

1. 以 `myblog-source` 作为唯一编辑真源继续写文
2. 决定哪些线上内容需要从旧静态层回填到源码层
3. 再选择合适窗口把新的构建产物正式发布上线

换句话说：

- 静态站点比对负责判断“有没有丢东西”
- 源码仓恢复负责判断“还能不能继续维护”

而真正的长期工作，必须回到源码仓里完成。
