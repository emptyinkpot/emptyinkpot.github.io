## 一、先给整套博客体系一个总认知

如果只用一句话描述这套博客体系，可以概括为：

**Markdown 负责内容，静态站点生成器负责把内容编译成网页，GitHub Actions 负责自动构建，GitHub Pages 负责发布到线上。**

在 Hexo 阶段，这条链路是：

```text
写文章 -> 放入 source/_posts -> Hexo 读取文章与主题 -> 生成 public/ 静态文件
-> 推送到 GitHub -> GitHub Actions 自动构建/发布 -> GitHub Pages 提供访问
```

在 Astro 阶段，这条链路变成：

```text
写文章 -> 放入内容目录 -> Astro Content Collections 校验与读取内容
-> Astro 构建静态站点 -> GitHub Actions 自动部署 -> GitHub Pages 提供访问
```

所以无论是 Hexo 还是 Astro，本质都没有变：

- 内容仍然优先存放为 Markdown
- 网站仍然优先以静态文件形式发布
- 仓库仍然是真正的内容与配置事实源
- 自动发布仍然依赖 GitHub Actions 与 GitHub Pages

## 二、最初这套博客是什么，以及下一步能做什么

在最早的起步状态里，这个博客就是一个标准的 Hexo 博客仓库：

- 文章写在 `source/_posts/`
- 独立页面写在 `source/about/` 等目录
- 站点总配置在 `_config.yml`
- 主题相关覆盖配置在 `_config.landscape.yml`
- `scaffolds/` 提供新文章模板
- `.github/workflows/pages.yml` 负责自动部署

当时最重要的认知有两个：

1. 写作位置必须明确，正文默认都应该进入 `source/_posts/`
2. 博客不是“本地看到就算发布”，而是要经过一整条构建与发布链路

如果是新站起步，后续通常可以继续做这些事：

- 补“关于”页、导航页、标签与分类体系
- 配自定义域名与 HTTPS
- 建搜索、评论、SEO、统计等能力
- 调整主题与内容组织方式
- 从 Hexo 逐步迁移到更灵活的 Astro 架构

## 三、Hexo 博客到底是怎么工作的

Hexo 博客本质是一个静态网站项目。

所谓静态网站，意思是：网页在发布前就已经生成完成。用户访问时，服务器直接返回已经构建好的 HTML、CSS、JavaScript 和图片，而不是在线实时拼装页面。

放到 Hexo 里，可以拆成四步：

1. 在本地写 Markdown 文章
2. Hexo 读取文章、Front Matter、站点配置与主题模板
3. Hexo 把它们渲染为 `public/` 下的静态文件
4. GitHub Pages 把这些静态文件发布到公网

其中两个核心角色一定要分清：

- `Hexo`：静态站点生成器，负责把内容变成网页
- `GitHub Pages`：静态网站托管服务，负责把网页放到公网

## 四、这个仓库里最重要的位置

无论是在 Hexo 阶段排错，还是迁移到 Astro，下面这些位置都属于必须熟悉的高频入口：

- `_config.yml`：站点主配置，比如标题、语言、网址、永久链接
- `_config.landscape.yml`：Hexo 主题覆盖配置，比如菜单和侧边栏
- `source/_posts/`：博客正文目录
- `source/about/`：独立页面目录
- `scaffolds/`：新建文章模板
- `.github/workflows/pages.yml`：自动部署工作流

后续升级到 Astro 后，虽然具体目录结构会变化，但这些概念本身不会消失，而是被映射成：

- 内容目录
- 站点配置
- 主题/布局组件
- 部署工作流
- 资源目录
- SEO/评论/搜索等外围能力

## 五、`source/_posts` 到底是什么

`source/_posts` 不是线上博客本体，也不是“放进去就自动上网”的魔法目录。它只是 Hexo 约定的**文章源文件目录**。

Hexo 会在构建时扫描这个目录，把其中符合规则的 Markdown 识别为文章，再结合 Front Matter、主题模板和站点配置生成页面。

因此需要建立几个非常重要的认知：

- `source/_posts/` 存的是**文章源文件**
- `public/` 才是 Hexo 构建后输出的静态网页结果
- 线上博客显示与否，不由“文件是否存在于本地”决定，而由“是否完成构建并发布”决定

## 六、一篇文章是怎么被识别的

一篇 Hexo 文章通常需要两个条件：

1. 文件放在 `source/_posts/` 中
2. 文件具备可被识别的 Front Matter

典型 Front Matter 例如：

```yaml
---
title: 我的第一篇文章
date: 2026-03-26 20:00:00
tags:
  - 建站
categories:
  - 博客
---
```

Hexo 在构建时会读取这些字段，决定：

- 页面标题
- 发布时间
- 标签与分类
- URL 生成方式
- 是否参与站点列表与归档

如果 Front Matter 不完整、格式不合法、日期异常，或者被标记成草稿，文章都有可能无法按预期上线。

## 七、放进 `source/_posts` 后，到底发生了什么

这是 Hexo 阶段最容易混淆的一段链路，必须拆开看：

### 第一步：进入博客源码

你写好的 Markdown 文件先进入博客仓库，也就是本地源码目录。

### 第二步：Hexo 构建它

Hexo 读取文章、读取主题、读取配置，把 Markdown 渲染成 HTML，并生成整个站点的静态文件到 `public/`。

### 第三步：推送到 GitHub

只有把仓库提交并推送到远端，GitHub 才能接手后续的自动构建与发布。

### 第四步：GitHub Actions 自动发布

工作流触发后，GitHub Actions 会执行安装依赖、构建站点、发布到 GitHub Pages 等步骤。

所以“自动发布”的真实含义不是“本地文件一保存就上网”，而是：

**只要代码推送到指定分支，远端工作流就会自动完成构建与发布。**

## 八、Hexo 是怎么把 Markdown 变成网页的

Hexo 在构建阶段大致会做这些事：

1. 扫描 `source/` 里的内容文件
2. 解析文章头部 Front Matter
3. 读取站点配置和主题模板
4. 将 Markdown 渲染为 HTML
5. 生成归档、标签、分类、首页等页面
6. 把结果输出到 `public/`

也就是说，Hexo 不只是“渲染一篇文章”，而是会在一次构建里生成整个站点。

## 九、为什么 Hexo + GitHub Pages 很适合个人博客

这套方案适合个人博客，原因主要有这些：

- 上手成本低
- Markdown 写作体验直接
- 仓库天然保留版本历史
- GitHub Actions 提供自动构建能力
- GitHub Pages 提供稳定的静态托管能力
- 整体可维护性高，且便于迁移

它非常适合内容优先、低运维、希望把文章长期版本化管理的人。

## 十、GitHub 实际免费提供了哪些资源

这部分是原有 Hexo 搭建文章中的重点，也必须保留，因为很多人会误以为“GitHub 免费帮我做了全部服务端能力”。其实不是。

GitHub 免费提供的主要是以下几类能力：

### 1. 仓库存储

用于保存博客源码、文章 Markdown、主题配置、工作流文件与历史版本。

### 2. GitHub Actions 构建算力

用于在推送后自动安装依赖、执行构建命令、产出静态文件并进入部署流程。

### 3. GitHub Pages 静态网站托管

用于托管最终生成的静态网站内容。

### 4. 默认域名

通常可以得到类似 `用户名.github.io` 的默认站点访问域名。

### 5. 自定义域名能力

可以绑定自己的域名，对外形成更正式的博客地址。

### 6. HTTPS 与加密访问

GitHub Pages 支持 HTTPS，这意味着默认就具备较好的基础访问安全性。

### 7. 自动发布链路

当仓库更新后，工作流可以自动重新部署，不需要手工上传网页文件。

### 8. 版本历史与回滚能力

因为整个站点都在 Git 中管理，所以文章、配置、主题变更都可以被追溯。

## 十一、这些资源的边界是什么

这部分同样很重要，因为它决定了你后续能否正确规划站点能力。

GitHub Pages 的边界主要在于：

- 它擅长托管静态资源，不是通用后端平台
- 它不适合直接承担数据库、登录系统、服务端渲染业务
- 动态能力一般需要借助第三方服务、前端 API 调用或者外部平台

因此评论、搜索、分析、表单、数据库等能力，往往都需要额外方案。

这也是后来迁移到 Astro 时，需要同时规划搜索、评论、SEO 与部署的一部分原因。

## 十二、从零搭建一个 Hexo 博客的完整步骤

把原有搭建文章中的实践链路合并后，可以归纳为下面这套流程：

### 1. 准备环境

- 安装 Node.js
- 安装 Git
- 确认本地可以运行 npm 命令

### 2. 创建 GitHub 仓库

建立一个用于承载博客源码的仓库。

### 3. 初始化 Hexo

创建 Hexo 项目，安装依赖，并生成基础目录结构。

### 4. 配置站点信息

在 `_config.yml` 中设置：

- 站点标题
- 描述
- 语言
- URL
- 永久链接规则

必要时再通过 `_config.landscape.yml` 调整主题配置。

### 5. 写第一篇文章

把文章写入 `source/_posts/`。

### 6. 本地预览

先在本地运行预览，确认页面可读、链接正常、格式正确。

### 7. 生成静态文件

执行构建，检查 `public/` 产物是否正确生成。

### 8. 配置 GitHub Actions 自动部署

通过工作流实现推送即自动部署。

### 9. 在 GitHub 中启用 Pages

确认 Pages 指向正确的 Actions/部署来源。

### 10. 提交并推送

把文章、配置、工作流一并推送到远端，让自动发布真正发生。

## 十三、这套博客现在的真实发布链路

这部分需要用更严格的方式记住：

1. 文章存在于本地源码仓库中
2. 文章被 Hexo 或 Astro 构建为静态站点内容
3. 本地构建只代表“本地能生成”
4. 推送到 GitHub 后，远端工作流才会开始自动构建/部署
5. GitHub Pages 最终对外提供线上访问

换句话说：

- 本地预览 != 线上已发布
- `source/_posts/` != 线上网站
- `public/` != 你日常应手工维护的长期事实源
- 推送代码 != 一定部署成功，仍要看工作流结果

## 十四、平时写博客的正确工作流

把几篇 Hexo 文章合并后，日常工作流可以收束成一套统一版本：

### 1. 新建文章

在 `source/_posts/` 中创建 Markdown 文件，或通过脚手架生成。

### 2. 写文章

补齐标题、日期、标签、分类等 Front Matter，并完成正文内容。

### 3. 本地预览

检查排版、代码块、图片、链接、分类标签、路径是否正确。

### 4. 本地构建检查

确保正式构建可以通过，不只是开发服务器能打开。

### 5. 提交并推送

把改动提交到仓库并推送到远端分支。

### 6. 等待线上自动发布

观察 GitHub Actions 是否成功，再确认 GitHub Pages 页面是否更新。

## 十五、哪些情况文章不会上线

这是原有发布流程文章中的重点排错清单，必须完整保留在总稿中。

### 情况 1：你只是放在本地，还没推送

本地文件不会自动同步到 GitHub，更不会自动上线。

### 情况 2：推送了，但 GitHub Actions 失败

代码已经进仓库，但远端构建失败，线上仍不会更新。

### 情况 3：文章是草稿

如果处于草稿态，构建时可能默认不发布。

### 情况 4：文章日期、路径或元数据写得不合理

例如 Front Matter 错误、日期异常、路径冲突、字段格式不对，都可能造成识别异常。

### 情况 5：Pages 配置没指向 GitHub Actions

即便仓库与工作流存在，如果 Pages 发布源没配对，线上也不会按预期更新。

## 十六、最容易记错的几个点

这部分是多篇文章重复强调的误区，我合并为一组统一提醒：

### 误区 1：`source/_posts` 等于线上博客

错。它只是文章源文件目录。

### 误区 2：`public/` 才是我该长期维护的地方

错。`public/` 是构建产物，真正应维护的是源文件和配置。

### 误区 3：我本地能看到，线上就一定能看到

错。中间还隔着提交、推送、工作流、Pages 发布。

### 误区 4：推送后一定马上可见

错。工作流和 Pages 都存在延迟或失败可能。

### 误区 5：建站就只是学几个命令

错。真正关键的是理解“内容源 -> 构建 -> 部署 -> 托管”的完整链路。

## 十七、为什么后来要从 Hexo 迁移到 Astro

随着内容和功能变多，Hexo 虽然仍然可用，但在内容建模、组件化、现代前端能力、扩展灵活性上开始显得不够理想。

迁移到 Astro 的主要动机包括：

- 希望保留静态站点的性能优势
- 希望拥有更现代的组件化开发体验
- 希望内容模型更清晰、可校验
- 希望更方便地接入搜索、评论、SEO 与后续功能增强
- 希望为长期演进打下更稳的结构基础

## 十八、Astro 重构的总体目标

原迁移文章把目标拆得比较清楚，这里统一收编：

### 1. 保留原有内容资产

迁移不是推倒重来，而是要把已有文章、分类、标签、链接资产平滑转过去。

### 2. 升级技术栈

让站点进入更现代的内容驱动与组件驱动体系。

### 3. 改善页面结构与可维护性

让首页、文章页、内容集合、SEO、评论、搜索都具备更清晰的边界。

### 4. 保持部署链路稳定

仍然使用 GitHub Actions + GitHub Pages 作为自动化发布底座。

## 十九、Astro 阶段的最终技术栈

原文中明确了 Astro 重构后的目标方向，可以概括为：

- Astro 作为静态站点框架
- 内容通过内容集合进行结构化管理
- 评论通过 giscus 接入
- 搜索通过构建后生成索引
- SEO 和部署继续纳入统一工作流

一个关键示例是内容集合 schema：

```typescript
const posts = defineCollection({
  type: 'content',
  schema: z.object({
    title: z.string(),
    description: z.string(),
    published: z.date(),
    updated: z.date().optional(),
    slug: z.string(),
    tags: z.array(z.string()),
    category: z.string(),
    series: z.string().optional(),
    featured: z.boolean().default(false),
    draft: z.boolean().default(false),
    cover: z.string().optional(),
    toc: z.boolean().default(true),
  }),
});
```

这个结构说明：迁移后的内容不再只是“能渲染就行”，而是开始被更严格地建模和校验。

## 二十、从 Hexo 到 Astro 的迁移阶段

把原文中的迁移步骤整合后，可以分成下面几个阶段：

### 阶段 1：项目初始化

- 初始化 Astro 项目
- 配置基础目录
- 确定内容、组件、布局与静态资源结构
- 安装所需依赖

### 阶段 2：内容迁移

- 把原 Hexo 内容迁移到新的内容目录
- 对 Front Matter 做字段对齐与清洗
- 保留 slug、标签、分类、时间等核心信息
- 修正资源路径、图片引用与历史链接策略

### 阶段 3：首页与文章页实现

- 实现首页信息组织
- 实现文章详情页
- 处理文章列表、分类、标签、目录等阅读结构

### 阶段 4：搜索与评论接入

- 构建后生成搜索索引
- 接入 giscus 评论系统

### 阶段 5：SEO 与基础设施

- 完成元信息、站点地图、基础 SEO 配置
- 完善静态资源和页面输出策略

### 阶段 6：部署配置

- 保持 GitHub Actions 自动化部署
- 适配 Astro 的构建命令与产物目录
- 确认 GitHub Pages 线上可访问

## 二十一、内容迁移时最重要的原则

无论从 Hexo 迁到 Astro，还是以后再做别的迁移，都要抓住这些原则：

### 1. 内容优先于框架

文章内容本身才是最核心资产，框架只是承载方式。

### 2. 元数据要完整迁移

至少要保留：

- 标题
- 日期
- slug
- 标签
- 分类
- 描述
- 封面与补充字段（如果已有）

### 3. 历史链接要尽量稳定

不要轻易打断旧链接体系，否则 SEO 与历史引用都会受影响。

### 4. 资源路径要统一清理

图片、附件、引用链接在框架迁移中最容易出错。

### 5. 迁移后要做功能验证

不能只看“能构建”，还要验证阅读链路、搜索、评论、SEO 与部署结果。

## 二十二、搜索与评论为什么要单独提

原 Astro 迁移文里，搜索和评论被单独拿出来，不是偶然，而是因为它们恰好代表了静态博客最常见的“准动态能力”。

### 1. 搜索

搜索通常不是服务端实时查数据库，而是通过构建阶段生成索引，再在前端加载索引完成检索。

这意味着：

- 搜索质量依赖内容索引是否完整
- 搜索更新依赖重新构建与重新部署
- 搜索方案应尽量与内容模型保持一致

### 2. giscus 评论

giscus 的接入重点在于配置项必须完整且真实可用，例如：

```astro
const repo = import.meta.env.PUBLIC_GISCUS_REPO;
const repoId = import.meta.env.PUBLIC_GISCUS_REPO_ID;
const category = import.meta.env.PUBLIC_GISCUS_CATEGORY;
const categoryId = import.meta.env.PUBLIC_GISCUS_CATEGORY_ID;
const isConfigured = Boolean(repo && repoId && category && categoryId);
```

这类配置说明一个原则：

**评论能力虽然运行在页面里，但它依赖的是外部服务配置，而不是博客仓库单独闭门完成。**

## 二十三、GitHub Actions 部署工作流在新旧体系中的共同作用

不管是 Hexo 还是 Astro，GitHub Actions 依然是自动构建与自动发布的关键中枢。

工作流的核心入口通常类似：

```yaml
on:
  push:
    branches: [main]
```

它的意义是：当主分支收到推送后，自动触发构建与部署流程。

在旧体系里，它负责构建 Hexo；在新体系里，它负责构建 Astro。真正变化的是构建命令与产物细节，不变的是自动化发布思想。

## 二十四、迁移完成后应该验证什么

原迁移文中已经给出验证方向，这里整合成统一清单：

### 功能验证

- 首页是否能正常展示
- 文章列表是否完整
- 文章详情页是否正常渲染
- 标签、分类、归档是否可访问
- 图片与静态资源是否正确加载
- 搜索是否能命中内容
- 评论是否正常显示或优雅降级

### 部署验证

- GitHub Actions 是否执行成功
- 部署产物是否正确上传
- GitHub Pages 是否可访问
- 历史页面链接是否仍尽量可用

### 技术升级验证

- 内容 schema 是否通过校验
- 构建过程是否稳定
- SEO 基础元信息是否完整

### 维护优化验证

- 新文章是否容易继续接入
- 站点结构是否清晰
- 配置入口是否单一
- 未来换主题、换样式、加功能是否更容易

## 二十五、这次整合后，对“建站”主题最值得保留的长期经验

把原几篇文章全部收束后，真正值得长期保留的经验主要有这些：

### 1. 先理解链路，再学命令

如果不理解“内容源 -> 构建 -> 发布 -> 托管”，命令记得再多也容易在排错时迷失。

### 2. 维护源文件，不维护产物

应长期维护的是 Markdown、配置、模板、工作流，而不是手工改构建产物。

### 3. 本地成功不代表线上成功

本地预览、本地构建、远端部署、线上可见，是四个不同阶段。

### 4. 自动化不是魔法，仍然要看事实源

工作流、Pages、环境配置、部署目标，任何一个点出错都可能让“自动发布”失效。

### 5. 框架可以换，内容体系不能乱

从 Hexo 到 Astro 的迁移说明：真正应该长期稳定的是内容组织、元数据质量和部署纪律，而不是对某个框架实现细节的依赖。

## 二十七、一句话收束

如果把这整个建站主题压缩成最后一句话，那就是：

**先把博客当成“内容源 + 构建链 + 发布链”的系统来理解，再去谈 Hexo、Astro、主题、搜索和评论，后面的每一次升级才会稳。**
