这篇文章不是讲某一个功能怎么做，而是把这段时间里实际踩过的坑、排错时最容易犯的错，以及后面应该固化成默认动作的经验收束成一份短文。

很多问题表面看起来只是一个小错误，但如果反复出现，往往说明工程纪律本身还不够稳。

## 核心问题总结

### 1. TypeScript 文件中的 HTML 转义字符问题

有一类问题不是业务逻辑错了，而是文件内容在流转过程中被错误转义了。

例如原本应该直接保留的代码或文本，被写成了 HTML 实体，最后会导致：

- 代码行为异常
- 页面渲染不符合预期
- 修改时肉眼不容易第一时间发现

这类问题说明：

- 不能只看“文件存在”
- 还要看“文件内容有没有被错误改写”
- 修改后必须检查关键片段是不是保持了原始语义

### 2. `index.html` 中的中文乱码问题

中文乱码通常不是一个孤立问题，它往往意味着：

- 文件编码不统一
- 输出链路里某一环对编码处理不正确
- 页面看起来能打开，但实际内容已经失真

这类问题最危险的地方在于：

- 页面表面“能显示”
- 但内容质量已经明显下降
- 如果不做细查，很容易把错误版本继续发布出去

所以只要涉及中文页面、模板文件、HTML 输出，就必须把编码检查当成正式验证的一部分。

### 3. 模块路径引用错误

路径错误是最常见、也最容易被低估的一类问题。

很多时候系统报错并不是核心逻辑有问题，而是：

- 引用路径写错了
- 目录层级判断错了
- 改目录后没有同步改引用
- 运行态真正读取的文件并不是以为的那一个

路径问题看起来“小”，但它会直接导致：

- 模块加载失败
- 运行结果偏离预期
- 排错方向被带偏

所以路径相关修改一定要逐级核对，不能凭印象写。

## 后面应该固化成默认动作的几点经验

### 1. 优先查看服务日志

遇到问题时，第一反应不应该是猜，也不应该是急着回滚，而应该先看日志。

日志的价值在于：

- 它更接近运行态事实
- 能帮助区分“表面现象”和“真正报错点”
- 能减少拍脑袋排查

如果不先看日志，后面很多改动都可能是在错误方向上继续消耗时间。

### 2. 谨慎修改文件

真正危险的不是“不会改”，而是“改得太快”。

尤其是在这些场景里，更要谨慎：

- 不完全确定文件是否是真正事实源时
- 不完全确定编码、语法、路径是否会被影响时
- 不完全确定当前改动会不会波及其他模块时

能先定位清楚的，就不要靠连续试错去撞结果。

### 3. 路径引用要仔细

路径问题必须一层一层确认，尤其要确认：

- 当前文件相对的是哪一层目录
- 运行时实际从哪个入口读到它
- 改目录后是否还有历史路径残留

这类问题如果只靠目测，很容易漏掉一个层级，最后导致判断全偏。

### 4. 不要频繁回滚

回滚不是不能做，但不能把它当成默认排错手段。

频繁回滚的问题在于：

- 容易把真正原因一起抹掉
- 容易让现场变得更难复盘
- 容易在还没理解问题前就重复制造新变量

更稳的做法是：

- 先保留现场
- 先看日志和实际报错
- 先缩小问题范围
- 再决定是否需要回滚

### 5. 验证要彻底

很多问题之所以反复出现，不是因为没有修，而是因为验证不完整。

所谓“验证彻底”，至少意味着：

- 不只看一个表面现象是否恢复
- 要检查相关链路有没有一起恢复
- 要确认这次修改没有引入新的副作用
- 要确认运行态和输出结果都符合预期

只有这样，才能避免“看起来修好了，过一会儿又坏”的循环。

## 一个很实用的检查思路

遇到类似问题时，可以按下面这个顺序快速自检：

```bash
# 检查文件内容是否有转义问题
# 检查页面文件是否存在编码异常
# 检查模块引用路径是否仍然正确
# 检查服务日志里最早的真实报错
# 检查修改后相关链路是否全部通过验证
```

这不是具体命令清单，而是一条排查顺序提醒：

先看内容是否被改坏，再看编码是否异常，再看路径是否错误，最后结合日志与完整验证来确认问题是否真正解决。

## 总结

如果把这篇短文压缩成一句话，那就是：

**排错时先看运行态事实，再谨慎修改，路径和编码都要细查，修完一定做完整验证，不要靠频繁回滚代替诊断。**
