这篇文章的目标，是把当前 OpenClaw 在 Windows 环境下已经验证过的网关启动、健康判定、浏览器联动停机、状态文件和日志路径等规则整理成一份可执行说明，避免以后再用旧心智去判断“到底算不算启动成功、到底怎样才算真正停掉”。

## 一、适用范围

本文只讨论当前已经验证通过的 Windows 本机启动链，不讨论历史旧链路，也不讨论理想化但尚未接入当前主流程的未来结构。

当前这份说明重点解决五件事：

1. 统一启动入口到底是什么
2. 启动成功到底以什么信号为准
3. 为什么不能再盯着 launcher 进程是否退出
4. 自动停机到底靠什么触发
5. 状态文件与日志为什么属于运行态事实

## 二、启动和停机的几个硬规则

### 1. 只认固定端口 `5000`

当前基线里，网关服务是否可以被复用、是否已经存在、是否成功起来，首先围绕固定端口 `5000` 判断。

这意味着：

- 启动前要先看端口能否复用
- 不要一会儿换端口、一会儿再猜服务地址
- 后续健康检查与外部验证也都应围绕这个固定端口进行

### 2. 网关是否真的启动成功，只以 `/health` 为准

这是最关键的一条。

不能再把这些信号当成最终判定标准：

- 某个脚本窗口有没有退出
- 某个 launcher 进程是不是还挂着
- 某个批处理文件是不是已经返回
- 浏览器有没有刚好打开

真正可靠的判定标准只有一个：

- `http://localhost:5000/health` 是否返回健康结果

### 3. 启动成功与否，不再盯着 launcher 进程退出与否

因为当前链路里，前台与后台已经不是简单的一次性阻塞关系。

实际情况更接近：

- 前台做受控拉起与短探测
- 后台继续等待服务真正健康
- 浏览器和 watcher 再接入后续运行链

所以看到 launcher 提前退出，不应立刻判定失败；看到 launcher 还挂着，也不代表服务一定健康。

### 4. 停机不再依赖 `openclaw gateway stop`

当前停机链已经不是过去那种“手动敲一个 stop 命令就等于全部收尾”的模型。

现在更强调：

- 受控浏览器窗口
- session watcher
- 状态文件同步
- 最后一个窗口关闭后的停机逻辑

### 5. 浏览器关闭自动停机，靠受控 session watcher，不靠前端 unload

这条规则必须记牢。

不能再把前端页面的 `unload` 或浏览器普通关闭行为，误认为是可靠停机机制。当前自动停机依赖的是受控 watcher 逻辑，而不是页面层的脆弱事件。

## 三、当前真实目录与职责链

当前已确认的实际脚本链包括：

- `start-openclaw-admin.ps1`
- `start-openclaw.bat`
- `openclaw-gateway-run.ps1`
- `openclaw-gateway-child.ps1`
- `openclaw-browser-launch.ps1`
- `openclaw-browser-watch.ps1`

把职责简单拆开，可以这样理解：

- `start-openclaw-admin.ps1`：处理需要管理员上下文的启动入口场景
- `start-openclaw.bat`：统一的主入口
- `openclaw-gateway-run.ps1`：负责网关启动主流程
- `openclaw-gateway-child.ps1`：承接后台子流程与受控启动链
- `openclaw-browser-launch.ps1`：负责浏览器拉起
- `openclaw-browser-watch.ps1`：负责浏览器会话监控与自动停机联动

## 四、规则 1：统一入口是 `start-openclaw.bat`

以后无论是人工执行还是让脚本再次封装，都应该以 `start-openclaw.bat` 作为统一入口。

原因很简单：

- 它是当前主链已经验证通过的事实源
- 绕开它去直接拼接其他脚本，最容易把主链打散
- 多入口并行只会让排错重新回到混乱状态

所以如果将来再新增别的启动方式，也应该是对统一入口的受控包装，而不是重新创造第二条主链。

## 五、规则 2：先判定能否复用 `5000`

启动前的第一步不是盲目拉起，而是先看当前 `5000` 端口是否已经被可复用的网关占用。

这一步的意义在于：

- 防止重复启动
- 防止把“已有健康服务”误判成“需要再起一个”
- 防止残留进程和僵尸状态把启动结果搅乱

如果端口已被正确服务占用，那么流程应优先考虑复用，而不是继续并发拉起第二份主启动流程。

## 六、规则 3：前台只做短探测，后台继续等健康

当前主链的设计，不是让前台永久阻塞等待，而是：

- 前台先发起启动
- 做短时间探测
- 后台继续等待服务真正健康
- 成功后进入浏览器与 watcher 联动阶段

这意味着“前台命令很快结束”本身并不说明失败，关键还是回到 `/health` 检查结果。

## 七、规则 4：启动时始终只允许一个主启动流程

这也是硬约束之一。

一旦出现下面这些行为，系统就容易重新进入混乱：

- 连续重复点击启动
- 同时从两个不同入口拉起
- 前一个流程没结束又手动补开一个
- 不检查端口和状态就再次执行启动

当前系统必须默认：

**同一时刻只允许一个主启动流程存在。**

## 八、自动停机的三条规则

### 规则 1：关闭“受控浏览器窗口”才会触发自动停机

不是任何浏览器窗口关闭都会触发停机，而是必须是当前启动链拉起并纳入 watcher 管控的受控窗口。

### 规则 2：只在最后一个受控窗口关闭后才停网关

如果同时存在多个受控窗口，那么关闭其中一个不应直接停掉整个网关。只有最后一个受控窗口关闭后，停机逻辑才应该成立。

### 规则 3：终止时同步写状态文件

停机不能只体现在进程层面，还必须体现在状态文件层面。否则下一次启动或排查时，很容易对当前真实状态产生误判。

## 九、启动与终止架构图

把当前链路简化成一张逻辑图，可以理解为：

```text
start-openclaw.bat
  -> openclaw-gateway-run.ps1
    -> openclaw-gateway-child.ps1
      -> 等待网关健康
      -> openclaw-browser-launch.ps1
      -> openclaw-browser-watch.ps1
        -> 监听受控窗口
        -> 最后一个窗口关闭后触发停机
        -> 写入状态文件并完成收尾
```

这张图最重要的价值，不是画得复杂，而是说明：

- 启动不是单脚本一次性动作
- 浏览器不是纯展示层，它参与了受控停机链
- watcher 是主流程的一部分，不是无关附件

## 十、状态文件规范

状态文件属于当前运行态事实源的一部分。

它至少承担这些职责：

- 记录当前是否在启动中、运行中或已终止
- 给后续脚本判断是否需要继续执行提供依据
- 给人工排查提供证据链

如果没有这层状态记录，很多现象都会变成“看起来像停了”“看起来像还在跑”的模糊判断。

## 十一、日志路径规范

日志同样不是附属品，而是判断当前主链是否健康的关键事实源。

后续排查时，日志至少可以帮助判断：

- 正常启动链是否走完
- 健康检查卡在哪一步
- 浏览器是否成功拉起
- watcher 是否开始工作
- 停机链是否真正执行到位

所以以后再排错，不能只盯着终端窗口，要同时看：

- `/health`
- 状态文件
- 日志输出
- 端口状态

## 十二、几种高频场景怎么判断

### 1. 正常启动

正常启动时，应该看到的是：

- 统一入口被调用
- 端口检查通过或正确复用
- 后台等待健康成功
- `/health` 返回正常
- 浏览器按受控流程拉起
- watcher 进入工作状态

### 2. 验证是否真的成功

验证标准不要再分散，统一按下面几项判断：

- `5000` 是否已被预期服务占用
- `/health` 是否返回健康
- 日志里是否有成功进入运行态的证据
- 状态文件是否与当前状态一致

### 3. 需要重启时怎么做

重启前，先确认：

- 当前端口状态
- 当前状态文件是否仍表示旧会话未结束
- 是否仍有受控窗口未关闭

不要在状态未清的情况下直接并发再起一套流程。

### 4. 正常关闭

正常关闭应该走当前受控浏览器与 watcher 联动逻辑，确保：

- 最后一个受控窗口关闭
- 网关停机链执行完成
- 状态文件更新完成
- 日志中可以看到对应收尾证据

## 十三、三个典型异常场景

### 场景 1：脚本执行了，但页面没出来

这时不要立刻断定网关没起，也不要立刻去改前端。

应优先检查：

- `/health` 是否已经正常
- 日志是否显示浏览器拉起阶段报错
- 受控浏览器启动链是否被本机环境拦住

### 场景 2：健康了，但 Key Guard 还是不通

这种情况说明：

- 网关健康不等于所有上层页面或保护链都已经正常
- 需要继续检查页面层、资源层或相关联的接入链

也就是说，`/health` 是必要条件，但不是所有外围能力都正常的充分条件。

### 场景 3：浏览器关了，但网关没停

这时首先不要回到旧认知里怀疑“是不是 stop 命令失效了”，而应重点检查：

- 关闭的是不是受控窗口
- 是否确实是最后一个受控窗口
- watcher 是否还在工作
- 状态文件和日志是否显示停机链中断

## 十四、这次已验证通过的结果

当前已经确认通过的基线包括：

- 统一入口应是 `start-openclaw.bat`
- 健康判断以 `/health` 为准
- 固定端口基线是 `5000`
- 浏览器关闭自动停机依赖 watcher，而不是前端 unload
- 停机链必须伴随状态文件同步更新

## 十二、一句话结论

如果把当前 OpenClaw 网关启停架构压缩成一句话，那就是：

**统一从 `start-openclaw.bat` 进入，只以 `5000` 上的 `/health` 判断是否启动成功，把浏览器 watcher、状态文件和日志都当成主链事实源，才能避免再次回到旧的误判方式。**
