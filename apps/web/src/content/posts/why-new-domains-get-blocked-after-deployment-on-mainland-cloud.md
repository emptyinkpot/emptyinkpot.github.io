这次踩坑最让人恼火的地方，不是不会部署，也不是服务起不来，而是：

- 服务器已经正常
- `Nginx` 已经正常
- 域名解析已经正常
- 站点页面也已经能在服务器本机返回
- 但新子域名还是被拦下来了

而且更让人困惑的是：

- `blog.tengokukk.com` 可以访问
- `www.tengokukk.com` 和 `openclaw.tengokukk.com` 却不行

这类问题如果只从“代码有没有写对”这个角度看，很容易越排越乱。因为真正卡住它的，根本不是站点代码，而是**国内云平台的备案与外部校验链路**。

## 一、这次真实发生了什么

当前这台服务器的真实情况是：

- 公网 IP：`124.220.233.126`
- 博客：`blog.tengokukk.com`
- 门户：`www.tengokukk.com`
- 服务入口：`openclaw.tengokukk.com`
- 博客静态站由 `Nginx` 托管
- `OpenClaw` 服务继续运行在 `5000` 端口

从工程层面看，这几条链路都已经搭好了：

- `www.tengokukk.com` 已经解析到服务器
- `openclaw.tengokukk.com` 已经解析到服务器
- 门户站已经部署到 `/srv/tengokukk-portal/site`
- `OpenClaw` 反向代理已经接到 `127.0.0.1:5000`
- `http://openclaw.tengokukk.com/health` 可以正常返回 `200`
- `http://www.tengokukk.com/` 也已经能返回门户站页面

也就是说，**站点部署链路本身其实已经成功了**。

真正卡住的是后面两件事：

1. 腾讯云/备案访问链路对新域名做了拦截
2. Let’s Encrypt 在签发 HTTPS 证书时，校验拿到的是备案拦截页，而不是你的真实站点页面

## 二、为什么看起来像“技术故障”，其实不是

这种问题最容易误导人的地方，在于它的表现非常像普通技术故障：

- 浏览器打不开
- `HTTPS` 签不下来
- 新域名访问异常
- `502`、`403`、连接重置、证书校验失败之类的症状混在一起

如果不把“工程问题”和“平台问题”拆开，就会一直在服务器里原地打转。

但这次实际排查下来，服务器内部其实是健康的：

- `Nginx` 正常监听 `80/443`
- 静态站点文件已经在正确目录
- `OpenClaw` 的 `5000` 端口也在监听
- 服务器本机访问 `127.0.0.1` 返回正常

所以问题不在：

- 代码
- 构建
- 静态产物
- 反向代理基本语义

问题在**外部世界访问这个域名时，到底看到了什么**。

## 三、为什么博客能访问，而新子域名被拦

这是这次最让人觉得“不合理”的地方。

看起来你走的步骤都一样：

- 加解析
- 配 `Nginx`
- 申请证书

但实际上，外部校验看到的东西并不一样。

### 1. 博客那条链路已经先成功过

`blog.tengokukk.com` 在更早的时候已经完成了：

- 解析到服务器
- 被 `certbot` 成功验证
- 证书成功签发

一旦这条链路已经成功，后面它就会继续按已经生效的证书和配置工作。

### 2. 新子域名是在后面才接入的

当 `www.tengokukk.com` 和 `openclaw.tengokukk.com` 后来再去申请证书时，Let’s Encrypt 去访问它们，拿到的却不是你的真实页面，而是类似这种结果：

- 备案拦截页
- 腾讯云 / DNSPod 的拦截返回

这就会导致：

- 证书申请失败
- `HTTPS` 无法建立

换句话说：

> 不是博客天生可以，新子域名天生不可以，而是博客当时先成功通过了那次校验，而后面两个子域名在校验时撞上了备案拦截。

## 四、为什么 HTTP 能通，HTTPS 却补不上

这次还有一个很典型的现象：

- `http://www.tengokukk.com/` 可以访问
- `http://openclaw.tengokukk.com/health` 可以访问
- 但 `https://www.tengokukk.com/` 和 `https://openclaw.tengokukk.com/` 不行

原因是：

- `HTTP` 访问只需要 DNS + `Nginx` + 站点内容
- `HTTPS` 还需要证书
- 证书签发又依赖外部校验

而一旦外部校验阶段被备案拦截页截走，证书签发就会失败。

所以你会看到一个很割裂的状态：

- 网站本身其实已经“能访问”
- 但想把它变成“正式 HTTPS 入口”时被卡住

## 五、这次真正学到的排障顺序

以后再遇到类似问题，最稳的排障顺序不是一上来就反复改配置，而是先分层判断。

### 第 1 层：服务本身有没有起来

先确认：

- 进程在不在
- 端口有没有监听
- 本机访问是否正常

### 第 2 层：反向代理有没有接对

再确认：

- `Nginx` 配置是否加载成功
- `Host` 转发是否正确
- 静态目录或代理目标是否返回预期内容

### 第 3 层：域名解析有没有到正确 IP

接着确认：

- 本机 DNS
- 服务器 DNS
- 公共 DNS

都能解析到同一个公网 IP。

### 第 4 层：外部校验到底拿到了什么

这是国内云场景里最容易被忽略的一层。

你必须确认：

- 外部浏览器访问拿到的是不是你的真实站点
- 证书校验拿到的是不是你的真实站点
- 有没有中间被备案拦截页替换掉

如果这一层错了，那前面三层再对，`HTTPS` 也还是上不去。

## 六、国内云上最让人崩溃的不是部署，而是“部署后半段”

如果只从工程角度看，这次其实没有特别复杂：

- 本地构建
- 上传静态产物
- 配置 `Nginx`
- 配 DNS
- 反向代理到后端

这些都不算难。

真正消耗人的，是后半段：

- 备案
- 接入放行
- 域名访问拦截
- 证书签发依赖外部校验

也就是说：

> 在中国大陆云服务器上，站点“部署完成”并不等于“正式上线完成”。

真正完整的上线链路应该理解成：

1. 工程部署完成
2. 域名访问放通
3. 备案 / 接入状态稳定
4. 证书签发成功
5. HTTPS 正式生效

少了后面几步，你得到的只是“半上线状态”。

## 七、当前这三个入口的最准确状态

这次最适合保留的真实口径是：

### 博客

- `https://blog.tengokukk.com/`
- 当前可正常访问
- 已有证书

### 门户

- `http://www.tengokukk.com/`
- 当前 `HTTP` 可访问
- `HTTPS` 待备案链路放通后补证书

### OpenClaw

- `http://openclaw.tengokukk.com/`
- `http://openclaw.tengokukk.com/health`
- 当前 `HTTP` 可访问
- `HTTPS` 同样待备案链路放通后补证书

## 八、以后怎么避免再被这种问题反复折腾

这次踩坑后，最值得记住的不是某个命令，而是一个部署心智：

### 1. 中国大陆云服务器的域名上线，要把备案算进主流程

不要把备案当成“最后顺手补一下”的附件。

### 2. 新子域名上线前，要预设它可能和旧域名不一样

哪怕：

- 主域名已经有内容
- 某个旧子域名已经能访问

也不能默认新加的子域名一定同样立即放行。

### 3. 证书问题先别急着怪 `certbot`

如果 `certbot` 报认证失败，要先怀疑：

- CA 外部校验拿到的是不是你的真实页面

而不是马上怀疑：

- `Nginx` 坏了
- 服务器坏了

### 4. 先让 HTTP 路径清楚，再谈 HTTPS 收尾

这次能比较快收口，就是因为我们先确认了：

- `www` 的 `HTTP` 已经能访问
- `openclaw` 的 `HTTP` 已经能访问

这样就知道：

- 工程链路已经通了
- 剩下只是证书与备案放行问题

## 九、这篇文章真正想留下的结论

如果你也在国内云服务器上部署站点，最容易误判的一句话就是：

> “我的网站都部署好了，为什么还是打不开？”

很多时候答案不是“你没部署好”，而是：

> 你把工程部分做完了，但平台合规和外部校验这半条链路还没让你过去。

这次最值得留下的一句话是：

> 在国内云环境里，域名上线不是单纯的 DevOps 问题，而是 DevOps + 合规接入的组合问题。

把这件事想清楚，后面很多看似离谱的问题，就不会再排错方向。
