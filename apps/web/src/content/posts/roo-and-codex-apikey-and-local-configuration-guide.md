这篇文章的目标，是把本机 VS Code 里 Roo 与 Codex 的 API Key 修改方式、活动配置真源判断方法，以及后续维护时最容易误判的点集中说明清楚。

它不是单纯回答“改哪个文件”，而是要把“为什么改起来会觉得麻烦”这件事本身讲明白。

## 1. 先说结论

以后如果要改 Roo 或 Codex 的 API Key，最重要的一句话是：

**先确认当前运行态真正读取哪份配置，再改对应事实源，最后做 reload 和验证。**

如果跳过这一步，最常见的结果就是：

- 文件明明改了，但运行时还是旧 key
- 以为 `.env.local` 是真源，实际它只是历史候选文件
- 以为某个目录里的配置就是当前生效配置，结果 VS Code 正在读另一份活动投影

## 2. 为什么 Roo 改 Key 会让人觉得比想象中麻烦

真正麻烦的地方，不是“多改了几个字符”，而是 Roo 的配置会涉及多层：

- VS Code 当前运行态
- Roo 扩展私有持久化目录
- 磁盘上的候选配置文件
- MCP 配置投影
- reload 之后才会生效的活动会话

如果不先分清这些层次，就会一直以为自己在改真源，实际却只是在改历史副本。

## 3. `.env.local` 到底是什么地位

在当前这台机器上，`.env.local` 不应该被当成唯一事实源。

更准确地说，它应被视为：

- 历史候选文件
- 辅助参考文件
- 某些时候用于补充信息的来源

但它不是你每次改 Roo key 时都必然要优先修改的唯一入口。

也正因为这一点，后面再改配置时，不能先入为主地把 `.env.local` 当成当前活动真源。

## 4. VS Code 运行态里的“活动配置”是什么意思

所谓 VS Code 运行态里的活动配置，指的是：

**当前这个 VS Code 会话中，扩展实际已经加载并正在使用的那份配置状态。**

这份状态可能来自：

- 扩展私有持久化目录中的配置投影
- 已经读入内存的活动存储
- 经过 reload 前后发生变化的运行会话

所以“磁盘上看到某个文件内容”不等于“当前运行态已经在用这份内容”。

这也是为什么很多时候改完还要 reload。

## 5. Codex 改 Key 的标准做法

Codex 的修改路径相对更清晰，但也仍然要遵守“先确认事实源”的原则。

### 1）先备份两个关键文件

在正式修改前，先备份：

- `auth.json`
- `config.toml`

这样做的原因很简单：

- 避免改错后没有回比基线
- 方便确认自己到底改了什么
- 出现异常时更容易判断问题出在 key 还是别的配置

### 2）改 `auth.json`

Codex 的核心鉴权信息优先看 `auth.json`。

如果目标是更换 API Key，这通常是第一优先入口。

### 3）改 `config.toml`

如果同时涉及：

- Base URL
- 供应商
- 模型
- 其他请求相关配置

那么需要一起检查并调整 `config.toml`。

### 4）不要误改 `.codex-global-state.json`

这个文件很容易被误判成“主配置文件”，但它不属于标准改 key 的核心入口。

### 5）改完后再验证

修改完成后，必须验证当前会话是否真的切到了新的鉴权与服务配置，而不是只在磁盘层面“看起来已经改好”。

## 6. Roo 改 Key 的标准做法

Roo 的正确做法，应按层次来。

### 第 1 层：先看 VS Code 运行态真源

先确认当前 VS Code 会话里的 Roo，到底正在使用哪份活动配置。

如果这一步没看清楚，后面改文件很可能全改错地方。

### 第 2 层：再检查 Roo 扩展私有存储

Roo 的关键配置和活动投影，往往在扩展私有持久化目录里，而不是某个看起来显眼的仓库路径。

这一步主要是为了确认：

- 当前活动配置投影是否存在
- 当前 key 信息是否已经在私有存储中生效
- MCP 与相关配置是否还是旧状态

### 第 3 层：把 `.env.local` 降级为候选参考

如果前两层已经明确活动真源，那么 `.env.local` 就不应该再被当成主入口。

它最多用于：

- 对照
- 留档
- 补充说明

而不是默认主改点。

### 第 4 层：最后刷新运行态

改完之后必须 reload，或者按需要执行受控重启扩展宿主。

否则很可能出现：

- 文件已经是新的
- 内存里还是旧的
- 表面像“没改成功”

## 7. 为什么“明明改了，为什么还是旧 key”会反复出现

常见原因主要有四类：

### 1）改错地方

改到了历史文件、备份文件、候选文件，而不是当前活动真源。

### 2）运行态没刷新

磁盘变了，但当前 VS Code 会话还在用旧缓存。

### 3）Base URL 没对齐

只改 key 不改 Base URL，或者 key 和 URL 对不上，最后看起来就像是 key 失效。

### 4）私有存储投影还是旧配置

即使你改了外层文件，只要当前活动投影还没更新，运行态仍然可能继续沿用旧值。

## 8. Base URL 为什么也是排障重点

很多人遇到鉴权失败时，第一反应只盯着 key。

但实际排查时，必须同时问：

- 当前请求发到哪里
- 当前 key 属于哪套服务
- 当前 Base URL 和供应商配置是否与 key 对应

否则很容易把“服务地址不匹配”误判成“key 本身有问题”。

## 9. Roo 的 MCP / Skill 配置应该怎么看

以后改 Roo 相关配置，不要只盯着 key 本身，还要知道当前 MCP / Skill 配置的入口在哪里。

更准确地说，要先判断：

- 当前真正该看的入口是哪一份
- 改 MCP 时要同步哪些位置
- 改完后为什么必须 reload

因为 Roo 的运行链不是“改一个文本文件即刻万事生效”，而是：

- 配置文件
- 活动投影
- 扩展运行态
- reload

共同决定的结果。

## 10. 后续维护建议（Roo）

以后维护 Roo 相关配置时，建议固定成下面这个顺序：

1. 先确认当前活动真源
2. 先备份关键配置与私有存储
3. 再改真正生效的配置入口
4. 必要时同步修正 MCP / Skill 相关投影
5. reload 后再做验证

不要再使用“我先顺手改一个最像的文件试试”的方式。

## 11. Codex 在本机 VS Code 中的标准改法

如果以后只是单纯修改本机 VS Code 中 Codex 的配置，最标准的思路是：

1. 先备份 `auth.json` 和 `config.toml`
2. 优先修改 `auth.json` 中的鉴权信息
3. 需要时同步调整 `config.toml` 中的 Base URL / 模型 / 供应商项
4. 不要把 `.codex-global-state.json` 当成主改点
5. 修改后重新加载会话并做实际调用验证

## 12. 最后收束成一句话

如果把这篇文章压缩成一句话，那就是：

**以后不管改 Roo 还是改 Codex，先分清当前 VS Code 运行态真正读哪份配置，再去修改那份事实源；否则改文件这件事本身并不难，难的是你根本没改到正在生效的那一份。**
