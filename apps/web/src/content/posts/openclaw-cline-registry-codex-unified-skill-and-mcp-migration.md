这篇文章记录的，不是一个“以后也许可以这么做”的设想，而是一轮已经落到本机、云端模块、VS Code 配置和 Codex 启动入口里的真实收口过程。

要解决的问题其实很具体：

> 技能和 MCP 明明已经迁移了，但不同代理、不同入口、不同配置文件还是各读各的，最后谁是真源、谁在生效、谁只是历史残留，根本说不清。

如果这个问题不收口，AI 工具链就会长期陷入三种非常典型的混乱：

- 内容已经迁了，但旧目录还在误导运行态
- 页面已经显示新模块，实际调用还指向旧路径
- 人以为“现在读的是统一真源”，实际上只是临时兼容层还没炸

这次真正要做的，不是“把目录挪一下”，而是把以下几件事同时做完：

1. 退役历史 `.cline`
2. 建立统一的云端 MCP / SKILL 模块
3. 让多个 agent 共享同一份真源
4. 让 Codex、VS Code、Cline 这些真实入口都改读新路径
5. 补齐桥接、自动部署、缓存一致性和验收链

## 一、这次的目标不是“能读到”，而是“稳定地读对”

很多迁移会卡在一个很迷惑人的中间态：

- 某个页面已经能显示新数据
- 某个脚本也能读取新目录
- 某次会话里手动试了一下也成功

然后大家就会误以为“已经迁移完成”。

但这类状态并不稳定，因为它只证明了一件事：

> 某一条路径在某个时刻碰巧能读到。

它没有证明更重要的东西：

- 以后新开的会话是不是还会这么读
- 其他代理是不是也会读到同一份内容
- 页面显示与真实运行态是不是一致
- 部署到云端后是不是还保留这个结论

所以这次收口的标准不是“读到了”，而是下面这四条同时成立：

### 1. 真源唯一

`云端MCP/SKILL` 模块内的 `library/skills` 与 `library/mcps` 成为唯一长期真源。

### 2. 运行入口真实改向

不是加一层兼容桥继续骗旧程序，而是把真实调用方改成直接读新路径。

### 3. 新会话默认接入

Codex 新开的会话、VS Code 的 MCP 配置、Claude Dev/Cline 的 MCP 配置，都要改成默认从新真源加载。

### 4. 自动部署和云端显示一致

本机改完之后，不需要靠记忆手动补传、补重启、补刷新；云端也必须能从工作站桥接看到真实运行态。

只有这样，才算“稳定地读对”。

## 二、为什么最后否定了兼容层

一开始最容易想到的做法，是把旧的 `~/.cline` 留着，然后把里面的 `skills` 和 `mcps` 用 junction 或软链接转到新模块。

这类方案短期很顺手，因为：

- 老程序不用改
- 目录看起来也还在
- 迁移成本似乎更低

但它有一个根本问题：

> 它把“历史路径”继续伪装成“现行路径”。

这会带来三个长期副作用。

### 1. 排障永远会被带偏

以后谁看到 `~/.cline`，都会自然认为：

- 这还是当前真源
- 出问题先去改这里
- 新模块只是一个镜像或副本

这样排障会一直绕回旧世界。

### 2. 配置治理永远收不干净

只要旧路径继续对外“看起来可用”，就会不断有人：

- 继续往里面加东西
- 继续在旧路径上做脚本
- 继续把新需求挂到旧入口上

最后的结果不是兼容，而是永久双真源。

### 3. 稳定性其实更差

兼容层并不能提高稳定性，它只是延迟暴露问题。

真正稳定的方式，反而是把真实入口全部改掉，让旧路径彻底退役。这样以后任何人再碰到旧路径，都会明确知道：

> 那不是当前入口了。

所以这次最后收口成了一个很明确的原则：

**不要兼容层，要改就彻底改。**

## 三、统一真源最后落在了哪里

这次最终收口后的结构非常明确：

- `projects/extensions/cline-registry/library/skills`
- `projects/extensions/cline-registry/library/mcps`

对应模块名称也从 `Cline Registry` 改成了更直白的：

> `云端MCP/SKILL`

这个模块不再只是展示页，而是已经承担了三层职责：

### 1. 内容真源

skill 与 MCP 的长期维护目录就在模块内部，不再依赖外部 `.cline`。

### 2. 共享注册表

它暴露出统一 API，多个 agent 可以读取 manifest、读取 skill payload、查看条目概览。

### 3. 管理入口

页面和后端已经能做新建、编辑、删除、重建快照，后续维护不必再回到历史目录里动手。

这意味着：

> 从“散落在本机目录里的私有配置”升级成了“有模块、有 API、有页面、有发布链的统一能力中心”。

## 四、为什么说现在能稳定读取，不是口头保证

用户问得最关键的一句，其实是：

> 你怎么保证以后可以稳定读取而不出岔子？

这次的答案不是“我会记住”，也不是“文档写清楚了”，而是把稳定性落实成了几条可验证的工程事实。

### 1. 真实调用方已经改读新真源

这一步非常关键。

我没有停留在“模块里有了副本”，而是直接改了实际调用方的配置，把原先指向 `C:\\Users\\ASUS-KL\\.cline\\mcps\\...` 的入口改成了模块内路径。

被真实改掉的入口包括：

- VS Code 的 `mcp.json`
- Claude Dev / Cline 的 `cline_mcp_settings.json`

这意味着：

- 旧路径不再是实际运行入口
- 新模块不是展示层，而是执行层的真源

### 2. Codex 的启动入口也被接到模块

如果只改 VS Code / Cline，而 Codex 仍然读自己老配置，那么“统一管理”依然是不成立的。

所以后面又补了一层：

- 把模块里的 skill 接到 `C:\\Users\\ASUS-KL\\.codex\\skills`
- 把模块里的 MCP 写进 `C:\\Users\\ASUS-KL\\.codex\\config.toml`

而且这里不是手工乱改，而是加了受控脚本和受管区块。

也就是说，新开的 Codex 会话会从同一套模块真源装载 skill 和 MCP。

### 3. 配置不是散改，而是受管写入

这一步很容易被忽略。

如果只是“把配置文件改到能跑”，后面很快还会漂移。

这次对 Codex 的接入做了一个更稳定的处理：

- 用固定脚本重建接入
- 在 `config.toml` 里使用受管区块标记
- 可重复执行
- 不依赖人手动一条条维护

这样以后即使要增减 MCP，也不会再退回“手工编辑 + 祈祷别写坏”的状态。

### 4. 云端模块与本机运行态已经打通

如果只是本机配置改了，云端页面还在显示旧 relay，那也会持续制造误判。

所以这次又把 `vscode-key-guard` 这条链一起修了：

- 本机 bridge 拉起
- SSH 反向隧道打通
- 云端页面优先使用工作站来源
- 如果当前运行身份不在候选列表中，就合成 runtime 条目

于是云端页面终于能正确显示：

- 当前运行的是本机 `C:\\Users\\ASUS-KL\\.codex`
- 当前身份是 ChatGPT 账号
- 不是服务器上的 relay key 在生效

### 5. 自动部署链已经修成“失败不自杀”

还有一个很现实的问题：

就算本机都改对了，如果自动部署 watcher 一次失败就退出，那云端迟早还是会回到旧版本。

这次也顺手把这条链修了：

- watcher 失败后不再直接退出
- 启动参数在带空格路径下可稳定运行
- 自动部署后的云端健康检查可真实反映结果

这意味着“今天改好了”不会因为下一次自动同步失败而悄悄回退成旧行为。

## 五、这次中间最容易误判的几个点

这轮收口不难做，难的是中间有很多“看起来已经好了”的假象。

### 1. 页面显示能骗人

一开始 `vscode-key-guard` 的页面已经能打开，但它显示的是服务器自己的 `/home/ubuntu/.codex`，不是本机工作站配置。

看起来页面没坏，实际上结论完全错。

这类问题说明：

> 页面能打开，不代表它显示的是对的运行态。

### 2. 桥通了，不代表前端逻辑已经升级

后来工作站桥接已经通了，云端接口也能读到本机账号，但页面仍然显示旧 relay。

原因不是桥坏了，而是：

- 云端跑的还是旧版模块
- 前端渲染逻辑还在以旧候选状态做高亮

这类问题很典型：

> 链路已经通，旧页面仍然可以继续制造错误结论。

### 3. 自动部署“看起来 idle”，其实 watcher 早死了

这次 `deploy-center` 有一段时间显示像还在工作，但其实 watcher 已经在一次失败后退出了，只留下一个过期状态文件。

所以后来又补了 watcher 活性判断：

- 进程是否还活着
- 状态文件是否过期
- 当前页面是否只是云端视角看不到本机状态

这一步修完后，才真正避免了“假 idle”误导。

### 4. “能读内容”和“自动注入成会话工具”不是一回事

一开始模块已经能通过 API 返回 skill 和 MCP 目录内容，但这并不等于我当前会话就自动拿到了这些工具。

后来真正完成“多代理共享”这件事，靠的是：

- 内容真源统一
- 运行入口改向
- Codex 启动接入

而不是仅仅做了一个展示或查询 API。

## 六、这次最后到底做到了什么

如果把这轮改造压缩成一句话，可以说是：

> 把历史 `.cline` 从一个“看起来还能用的旧入口”，彻底收口成了 `云端MCP/SKILL` 这个统一真源，并让 VS Code、Cline、Codex 与云端页面都围绕它运转。

更具体一点，最后落地的是这些结果：

### 1. `.cline` 已经退役

旧目录不再作为运行真源，也不再保留兼容桥作为长期方案。

### 2. 模块内 `library/` 成为唯一现行来源

后续 skill 和 MCP 的维护，都应该只发生在这个模块里。

### 3. 多个 agent 可以共享同一目录与 API

这意味着技能说明、注册表、管理页和后续能力扩展都不再是分散的。

### 4. Codex 新会话会从统一真源加载

这里很重要的一点是：

它影响的是之后新开的 Codex 会话，而不是已经在运行的旧会话。

### 5. 云端显示与本机运行态已经基本对齐

不再是页面显示一套、真实运行另一套。

## 七、这轮改造对以后有什么价值

这次工作的价值，不只是“把目录迁过去了”。

它真正建立的是一套以后还能继续扩张的治理边界。

### 1. 真源终于能明确写成一句话

以后再有人问：

> skill 和 MCP 现在到底该改哪？

答案终于可以非常干净：

> 改 `云端MCP/SKILL` 模块里的 `library/skills` 和 `library/mcps`。

### 2. 新能力可以从模块往外扩，不必继续在历史目录上打补丁

后面无论是：

- 增加白名单 MCP 代理执行
- 做技能上传与编辑入口
- 做代理启动时的远程注入

都应该以这个模块为中心继续扩展，而不是回头修补 `.cline`。

### 3. 稳定性从“记忆型”变成“结构型”

以前的稳定，依赖的是谁记得哪个路径、谁记得要 reload、谁记得页面其实显示的是另一台机器。

现在的稳定，更多来自：

- 统一真源
- 统一入口
- 自动部署
- 云端桥接
- 受控配置写入

也就是说，稳定性不再主要依赖人脑。

## 八、最后的结论

这次真正收口的，不是某个目录，也不是某个脚本，而是一个长期反复出现的工程问题：

> 当 skill、MCP、页面、代理启动入口和云端运行态来自不同地方时，怎样把它们统一成一个可长期维护、可多人共用、可自动部署、可验证的系统。

这次的答案最终非常明确：

- 不保留长期兼容层
- 不让旧路径继续伪装成现行入口
- 不接受多真源并存
- 把模块内 `library/` 变成唯一事实源
- 把真实调用方逐个改到新入口
- 把 Codex 启动也纳入统一接入
- 把桥接、自动部署和页面显示一起修到一致

如果只看目录变化，这轮工作好像只是“从 `.cline` 搬到一个模块里”。

但如果从工程治理角度看，它真正完成的是：

> 把一堆零散、历史、局部有效的 AI 配置，升级成了一套有真源、有 API、有页面、有自动化、有运行态接入的统一能力中心。

这才是这轮迁移真正值钱的地方。
