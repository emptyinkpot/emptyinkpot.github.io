这篇文档不是某个业务的局部设计说明，而是一份可适用于绝大多数项目的 **文件与模块开发规范总纲**。

它不绑定某个产品名，也不依赖某种前端框架、后端语言或部署方式。

它要解决的是更根本的问题：

- 工程为什么会越做越乱
- 模块为什么会越拆越耦合
- 文件为什么会越写越难找
- 需求为什么一改就牵一大片
- 团队和 AI 为什么总在不同目录里重复造轮子

这份规范的目标，就是把工程从“目录堆叠 + 共享失控 + 依赖穿透 + 状态扩散”的混合体，收敛成一套 **低耦合、高内聚、边界清晰、可持续演进的模块化工程体系**。

为了便于执行，这篇文章会把规范拆成以下部分：

1. 项目定位
2. 核心设计目标
3. 架构分层模型
4. 顶层目录规划
5. 模块标准结构规范
6. 平台内核与公共层规范
7. 文件组织规范
8. 路由、接口与依赖边界
9. 模块声明模型
10. 数据、配置与运行态规范
11. 测试、观测与质量门禁
12. 迁移与重构原则
13. AI 协作与自动化约束
14. 示例模板
15. 最终硬约束

## 1. 项目定位

一个可长期维护的现代工程，不应该是“很多页面 + 很多脚本 + 很多工具函数”堆在一起的集合。

它应该被定义为：

> **模块化单体优先、边界清晰、声明驱动、运行态外置的工程平台**

这里的“平台”不一定意味着它很大，而是说它具备以下特征：

- 顶层有统一的工程规则
- 模块是主要组织单位，而不是文件夹偶然分组
- 每个模块对自己的代码、配置、状态、接口负责
- 公共层只保留真正公共的最小集合
- 运行态与源码分离
- 目录结构能够反映职责，而不是仅反映技术栈

一个成熟项目最终通常由三层组成：

```text
平台层
├─ 极小内核 / 公共基础设施
├─ 构建与质量规则
└─ 模块加载与统一约束

模块层
├─ 业务模块
├─ 能力模块
└─ 集成模块

运行层
├─ cache / logs / temp / state
└─ 模块私有运行数据
```

这一定义很重要，因为它决定了工程不再追求“到处抽共享”，而是追求：

- 平台克制
- 模块闭环
- 声明驱动
- 依赖单向
- 状态收口
- 规则可校验

## 2. 核心设计目标

整个架构只追求八件事。

### 2.1 模块闭环

每个模块必须尽量自己收口自己的：

- 页面或视图
- 应用服务
- 核心领域逻辑
- 数据访问
- 对外契约
- 测试
- 文档

一个模块不是“几个页面 + 一个 service 文件”的松散组合，而是一个可理解、可测试、可替换的最小业务单元。

### 2.2 平台克制

顶层平台只承载底座能力，不承载业务实现。

平台层越胖，模块就越薄；平台层越像杂物间，模块边界就越模糊。

因此平台必须克制，只保留：

- 工程配置
- 统一日志、配置、鉴权、数据库、消息、HTTP 封装
- 测试底座
- 观测底座
- 公共类型与协议

### 2.3 边界清晰

每个模块都必须有明确的：

- 模块名
- 目录所有权
- 页面前缀
- API 前缀
- 配置归属
- 状态归属
- 依赖边界
- 对外暴露面

没有边界的模块，不叫模块，只叫“按目录放了一堆代码”。

### 2.4 高内聚

高内聚不是“代码写得近”，而是“变化原因相同的代码放在一起”。

因此一个模块内部应该围绕单一业务能力组织，而不是围绕技术类型平铺：

- 不要把全站所有 `service` 堆一起
- 不要把所有弹窗、所有 hooks、所有 utils 堆一起
- 不要把所有 SQL、所有 DTO、所有 schema 放成跨模块公共池

### 2.5 低耦合

低耦合不是没有依赖，而是：

- 依赖方向稳定
- 依赖面尽量窄
- 依赖经由契约而不是内部实现
- 模块之间互相不知道太多细节

模块 A 想用模块 B，不应直接摸进 B 的内部目录，而应依赖 B 对外暴露的稳定接口。

### 2.6 声明驱动

页面、导航、权限、插件、任务、运行目录、模块依赖都应该优先由声明驱动，而不是散落在各处手改。

声明驱动的价值在于：

- 更容易校验
- 更容易自动化
- 更容易让 AI 和人协作
- 更容易迁移和重构

### 2.7 运行态外置

源码仓库只存“可追踪、可审查、可版本化”的内容。

运行时产生的内容，例如：

- 缓存
- 日志
- 临时文件
- 本地状态
- 浏览器数据
- 上传中间产物
- 模型文件
- 索引文件

都必须外置，不得混入源码主树。

### 2.8 机器可校验

架构规则不能只靠文档和自觉，必须能被工具持续验证，例如：

- ESLint import boundary 规则
- `dependency-cruiser`
- `Semgrep`
- `jscpd`
- `Knip`
- 单测与契约测试
- 自定义目录检查脚本

## 3. 总体架构分层

推荐采用以下通用分层：

### 3.1 平台壳层

平台壳层负责统一入口和整体编排，但不承载具体业务。

它的职责通常包括：

- 应用入口
- 全局布局
- 导航壳
- 路由挂载
- 模块装配
- 全局错误边界
- 全局主题或设计系统注入

平台壳只做编排，不做业务判断。

### 3.2 平台内核层

内核层只放平台底座，不放业务能力。

允许内容：

- `config`
- `db`
- `http`
- `auth`
- `queue`
- `events`
- `obs`
- `types`
- `testing`
- `runtime`

禁止内容：

- 任意业务模块私有逻辑
- 某个业务的页面实现
- 某个业务的 DTO 细节
- 某个业务的数据库查询细节
- 某个业务专属工具函数

### 3.3 模块层

所有业务能力都应在模块层平行存在。

模块通常分三类：

| 类型 | 说明 |
| --- | --- |
| 业务模块 | 直接面向具体业务域，例如订单、内容、项目、用户、计费 |
| 能力模块 | 提供跨业务能力，例如搜索、通知、文件、AI、审计 |
| 集成模块 | 对接外部系统，例如支付、IM、第三方开放平台 |

### 3.4 运行层

运行层统一外置，按模块隔离。

```text
.runtime/
└─ modules/
   ├─ content-center/
   │  ├─ cache/
   │  ├─ logs/
   │  ├─ state/
   │  └─ temp/
   └─ search/
      ├─ cache/
      ├─ index/
      └─ logs/
```

运行层的核心原则：

- 源码与状态分离
- 公共运行资源与模块私有资源分层
- 允许清理运行态，不影响源码

## 4. 顶层目录规划

最终顶层只允许存在少数高价值目录。

```text
project-root/
├─ apps/
├─ modules/
├─ kernel/
├─ docs/
├─ tools/
├─ tests/
├─ .runtime/
├─ package.json
├─ tsconfig.base.json
├─ .eslintrc.cjs
├─ dependency-cruiser.cjs
└─ ARCHITECTURE.md
```

说明：

- `apps/` 用于承载平台壳、控制台壳、BFF、管理端等应用入口。
- `modules/` 用于承载业务模块、能力模块、集成模块。
- `kernel/` 用于平台级基础设施。
- `docs/` 用于规范、决策记录、迁移计划。
- `tools/` 用于迁移、生成、校验、发布脚本。
- `tests/` 用于跨模块、跨应用、端到端测试。
- `.runtime/` 用于运行期状态。

### 4.1 顶层明确禁止

顶层不应长期出现以下目录：

- `misc/`
- `temp-src/`
- `new/`
- `old/`
- `backup/`
- `test2/`
- `demo-final/`
- `public-copy/`
- `shared-everything/`

这些目录名本质上是在告诉后来者：

- 这里没有正式边界
- 这里没有归档策略
- 这里没有收敛计划

### 4.2 非正式目录处理

样例项目、实验目录、临时验证代码不得与正式模块并列。

处理方式只有三个：

- 迁出到 `sandbox/` 或 `examples/`
- 归档到明确的 `archive/`
- 直接删除

## 5. 模块标准结构规范

一个标准模块推荐长这样：

```text
modules/
└─ content-center/
   ├─ module.json
   ├─ README.md
   ├─ package.json
   ├─ src/
   │  ├─ contracts/
   │  ├─ application/
   │  ├─ domain/
   │  ├─ infrastructure/
   │  ├─ interface/
   │  └─ index.ts
   ├─ tests/
   │  ├─ unit/
   │  ├─ integration/
   │  └─ contract/
   └─ docs/
      ├─ decisions/
      └─ changelog.md
```

### 5.1 每层职责

#### `contracts/`

对外契约层，放稳定的暴露面，例如：

- DTO
- 输入输出 schema
- 事件定义
- service interface
- 类型声明

这层是模块间协作的首选入口。

#### `application/`

应用层，负责用例编排，不写底层细节。

它应该处理：

- 用例流程
- 权限检查调用
- 事务编排
- 领域对象协作
- 调用仓储接口
- 发布领域事件

它不应该直接知道数据库表细节或页面组件实现。

#### `domain/`

领域层，放业务核心规则。

例如：

- 实体
- 值对象
- 聚合
- 领域服务
- 领域规则
- 状态机

这层是高内聚的核心，变化应围绕业务本身，而不是框架本身。

#### `infrastructure/`

基础设施层，负责技术落地。

例如：

- Repository 实现
- ORM 映射
- 第三方 SDK 封装
- 消息中间件适配
- 文件存储实现
- 缓存实现

这层最容易污染业务边界，因此必须只实现协议，不反向主导业务结构。

#### `interface/`

接口层，负责对外接入。

例如：

- HTTP controller
- RPC handler
- CLI command
- 页面路由装配
- 表单适配器
- ViewModel 转换

接口层不应该直接写复杂业务逻辑。

### 5.2 模块导出原则

模块必须有唯一稳定出口，例如：

```text
modules/content-center/src/index.ts
```

禁止其他模块直接引用以下内部路径：

- `modules/content-center/src/domain/...`
- `modules/content-center/src/infrastructure/...`
- `modules/content-center/src/interface/...`

允许的方式应该是：

```text
@modules/content-center
@modules/content-center/contracts
```

### 5.3 模块命名规范

模块命名建议遵守：

- 业务名词优先
- 单数或集合表达保持一致
- kebab-case 命名
- 不使用含糊缩写

推荐：

- `content-center`
- `user-profile`
- `billing`
- `file-storage`
- `notification-center`

不推荐：

- `common2`
- `temp-biz`
- `manager-final`
- `all-service`
- `utils-pro-max`

## 6. 平台内核与公共层规范

很多项目的腐化都始于“公共层失控”。

因此必须给 `kernel/` 和公共层立严格规则。

### 6.1 `kernel/` 只允许放什么

只允许放真正全局且稳定的底座：

- 配置读取
- 日志与链路追踪
- 时间、ID、错误模型
- HTTP client / server 基础封装
- 数据库连接与事务基座
- 事件总线基础能力
- 权限框架壳
- 测试基建

### 6.2 `kernel/` 明确禁止什么

- 任何业务枚举
- 任何业务查询逻辑
- 任何业务页面
- 任何业务表单
- 任何“为了复用而提早抽象”的半成品
- 任意模块的私有 helper

### 6.3 公共代码进入 `kernel/` 的前提

一段代码要进入公共层，至少满足以下条件中的全部：

1. 已被两个以上模块独立需要
2. 语义稳定，不依赖某个业务命名
3. 抽出去后不会让调用方知道更多内部细节
4. 可以被单独测试和单独文档化

否则宁可在模块内保留局部重复，也不要把错误抽象放到公共层。

### 6.4 允许重复，禁止错误共享

这是大多数团队最难接受但最关键的一条：

> **短期可控重复，通常优于长期污染共享层。**

复制一小段简单逻辑并不可怕；把业务耦合塞进公共层，后续才最贵。

## 7. 文件组织规范

目录体现边界，文件体现职责。

### 7.1 单文件职责原则

一个文件只负责一类变化原因。

例如：

- 一个 controller 文件负责一个接口组
- 一个 use case 文件负责一个业务动作
- 一个 schema 文件负责一组稳定输入输出定义
- 一个组件文件负责一个清晰 UI 单元

不建议出现：

- 一个文件同时写 controller + service + repository
- 一个文件塞进十几个不相关 helper
- 一个页面文件同时处理取数、状态机、复杂校验、埋点和样式决策

### 7.2 文件长度建议

没有绝对行数标准，但建议遵循以下经验：

- 100 行以内：通常最容易维护
- 100 到 250 行：可以接受，但要检查是否可拆
- 250 行以上：大概率承担了过多职责
- 500 行以上：通常意味着结构已经失控

### 7.3 命名原则

文件命名要表达“角色 + 领域”，而不是“开发过程状态”。

推荐：

- `createArticle.useCase.ts`
- `ArticleRepository.ts`
- `article.schema.ts`
- `ArticleListPage.tsx`
- `publishWorkflow.service.ts`

不推荐：

- `newService.ts`
- `final.ts`
- `aaa.ts`
- `util.ts`
- `handle.ts`
- `testok.ts`

### 7.4 `utils/` 控制原则

`utils/` 不是垃圾桶。

只有以下两类内容适合放工具目录：

- 稳定、纯函数、无业务语义的通用工具
- 明确属于某个模块的局部工具

建议使用两级策略：

```text
kernel/utils/
modules/order/src/shared/utils/
```

禁止建立全局巨型 `utils/` 再让全项目随意引用。

### 7.5 组件文件规范

对于前端工程：

- 页面组件负责装配，不负责全部业务逻辑
- 可复用组件应尽量纯展示
- 模块私有组件优先留在模块内
- 跨模块共享组件必须先抽象出稳定设计语义，而不是只因为两个地方长得像

## 8. 路由、接口与依赖边界

### 8.1 路由所有权规范

每个页面前缀必须有唯一模块所有者。

例如：

- `/articles/*` 归 `content-center`
- `/projects/*` 归 `project-hub`
- `/billing/*` 归 `billing`
- `/admin/users/*` 归 `user-admin`

禁止多个模块共同维护同一路由树。

### 8.2 API 所有权规范

每个 API 前缀也必须有唯一归属。

例如：

- `/api/articles/*` 归 `content-center`
- `/api/files/*` 归 `file-storage`
- `/api/search/*` 归 `search`

如果一个接口需要跨模块协作，应由“拥有该场景的模块”负责发起编排，而不是建立匿名公共接口。

### 8.3 依赖方向规范

通用依赖方向应保持为：

```text
interface -> application -> domain
                     -> contracts
infrastructure -> domain / contracts
```

更关键的约束是：

- 接口层可依赖应用层
- 应用层可依赖领域层与契约层
- 基础设施层实现契约，但不要反向支配应用层
- 模块外部只能依赖模块公开契约

### 8.4 模块间协作规则

模块间协作只允许通过以下方式之一：

- 调用公开 service interface
- 调用公开 use case facade
- 使用公开 DTO / schema
- 订阅公开事件
- 访问明确暴露的查询接口

明确禁止：

- 跨模块深层导入内部文件
- 直接读写别的模块私有表结构细节
- 复制别的模块内部逻辑再魔改
- 共用一份隐式全局状态

### 8.5 反向依赖与环依赖

以下情况必须视为结构错误：

- 低层依赖高层
- 业务模块双向互依赖
- 模块 A 依赖 B，B 再依赖 A
- 模块间通过共享工具层间接环依赖

发现环依赖时，优先处理方式：

1. 提取稳定契约层
2. 引入领域事件或消息机制
3. 把双向流程改成单向编排
4. 重画模块边界，而不是继续打补丁

## 9. 模块声明模型

建议每个模块具备一份声明文件，例如：

```json
{
  "name": "content-center",
  "type": "business",
  "owner": "team-content",
  "description": "Manage article, note and page content lifecycle",
  "routePrefix": ["/articles", "/notes"],
  "apiPrefix": ["/api/articles", "/api/notes"],
  "dependencies": ["search", "file-storage"],
  "exports": ["contracts", "facade"],
  "runtime": {
    "cache": true,
    "logs": true,
    "state": true
  }
}
```

### 9.1 为什么需要声明文件

因为声明文件可以承载：

- 模块元信息
- 路由归属
- API 归属
- 依赖清单
- 对外暴露面
- 运行态需求
- 所有权归属

它既是文档，也是自动化检查输入。

### 9.2 声明驱动的范围

可由声明驱动的内容包括：

- 导航注册
- 路由注册
- 页面菜单
- 权限点清单
- 定时任务注册
- 模块加载顺序
- 运行目录初始化
- 构建检查

## 10. 数据、配置与运行态规范

### 10.1 配置分层规范

配置必须分层，不允许把所有配置都塞进一个全局文件。

建议分为：

- 平台全局配置
- 模块级配置
- 环境变量
- 运行期动态配置

每层配置都要明确：

- 谁负责维护
- 作用范围是什么
- 是否允许热更新
- 是否可被版本控制

### 10.2 数据访问规范

数据访问必须在模块边界内收口。

推荐方式：

- 模块维护自己的 repository
- 跨模块查询通过契约接口暴露
- 避免多模块直接拼接彼此表结构细节

### 10.3 事务边界规范

事务应由应用层用例统一控制，而不是散落在控制器、repository 和脚本中。

原则：

- 一个业务动作，一个明确事务边界
- 不跨模块隐式持有事务
- 不把事务传播成全局上下文魔法

### 10.4 运行态目录规范

运行态建议统一外置，例如：

```text
.runtime/
├─ app/
│  ├─ logs/
│  ├─ cache/
│  └─ temp/
└─ modules/
   ├─ content-center/
   └─ search/
```

明确禁止：

- 模块源码目录下长期落日志
- 把缓存、上传产物、索引文件提交进 git
- 在 `src/` 下存运行期状态

## 11. 测试、观测与质量门禁

### 11.1 测试分层

建议至少分三层：

- 单元测试：验证领域规则与纯逻辑
- 集成测试：验证模块内部协作
- 契约测试：验证模块对外接口稳定性

需要跨模块验证时，再加：

- 端到端测试
- 场景回归测试

### 11.2 测试归属原则

测试尽量跟着模块走。

- 模块私有规则测试放模块内
- 跨模块场景测试放顶层 `tests/`
- 平台底座测试放 `kernel/testing`

### 11.3 日志与观测规范

日志必须具备上下文，而不是只打一串字符串。

至少建议包含：

- 时间
- 模块名
- 请求或任务 id
- 用户或主体 id
- 动作名
- 成功失败状态
- 错误码或错误类型

### 11.4 质量门禁建议

工程至少应有以下自动检查：

- Lint
- Type check
- 单测
- import boundary 校验
- 环依赖检查
- 重复代码检查
- 未使用代码检查
- 关键模块契约校验

如果项目重要性较高，再加：

- API schema diff
- 迁移脚本回归校验
- 快照或视觉回归

## 12. 迁移与重构原则

迁移不是复制，重构不是换目录名。

### 12.1 迁移即清理

迁移完成后必须处理旧资产：

- 旧目录删除
- 旧路由删除或重定向
- 旧副本删除
- 旧别名删除
- 旧脚本下线

长期双份共存通常意味着未来一定失真。

### 12.2 重构顺序建议

建议按以下顺序推进：

1. 先定模块边界
2. 再定目录结构
3. 再提公开契约
4. 再移动实现
5. 最后接自动化检查

不要一上来就全局搜索替换和大搬家。

### 12.3 兼容策略

如果必须过渡共存，必须显式声明：

- 哪份是真源
- 哪份是兼容层
- 哪天删除
- 谁负责迁移完成

没有截止日期的兼容层，最后都会变成正式层。

## 13. AI 协作与自动化约束

当 AI 参与工程开发时，更需要结构规则。

### 13.1 AI 必须遵守的边界

- 不得随意跨模块深层引用内部文件
- 不得把业务逻辑塞进 `kernel/`
- 不得新建含糊目录承载临时实现
- 不得把运行态文件写入源码目录
- 不得复制旧模块再改名当新模块

### 13.2 人与 AI 的统一事实源

团队与 AI 应共享同一套结构事实源：

- 架构总纲
- 模块声明文件
- lint / dependency 规则
- 目录树与模板
- 自动检查脚本

### 13.3 让规则可被执行

理想状态不是“AI 看过文档”，而是：

- 新模块从模板生成
- import 越界直接报错
- 环依赖自动阻断
- 模块声明缺失直接拒绝合并
- 运行态目录写错位置直接告警

## 15. 最终硬约束

下面这些约束建议视为红线。

### 15.1 顶层红线

- 顶层只允许正式目录
- 不允许长期临时目录并存
- 平台壳、模块层、运行层必须分离

### 15.2 模块红线

- 模块必须有唯一名字、唯一职责、唯一边界
- 模块间不得深层导入内部实现
- 模块必须有稳定对外出口
- 模块必须能独立测试

### 15.3 公共层红线

- `kernel/` 不承载业务逻辑
- 公共层只收真正稳定、跨模块复用的基础能力
- 不允许把“未来也许会复用”的代码提前塞进公共层

### 15.4 文件红线

- 单文件不得混装多种职责
- `utils/` 不得成为垃圾桶
- 不允许 `final`、`new`、`temp`、`copy` 这类状态型命名长期存在

### 15.5 运行态红线

- 运行态不得混入源码主树
- 日志、缓存、状态、临时文件必须外置
- 不允许把运行产物提交进版本库

### 15.6 演进红线

- 新需求优先进入已有模块边界，而不是新建“补丁目录”
- 边界错误先修边界，再写功能
- 迁移完成必须清旧，不允许长期双份真源

## 结语

真正接近大厂工程思路的模块化，并不是把目录切得很碎，也不是把名词写得很高级。

它的核心一直只有几条：

- 平台克制
- 模块闭环
- 高内聚
- 低耦合
- 契约协作
- 声明驱动
- 规则可校验

当一个工程真正按这套方式演进时，目录会更稳定，依赖会更单纯，需求变更的影响面会更可控，人和 AI 也更容易在同一套结构规则下协作。

这才是模块化真正的价值：

> **不是为了“看起来规整”，而是为了让工程在持续迭代中依然保持可理解、可修改、可迁移。**
