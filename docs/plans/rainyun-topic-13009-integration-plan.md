# Rainyun 话题 13009 整合计划（项目化落地）

更新时间：2026-03-31

来源话题：`https://forum.rainyun.com/t/topic/13009?u=azzzw`

## 1. 目标定义

将话题中展示的“可配置个人主页 + 后台管理 + 一键跑路能力”整理为本仓库可持续维护的功能规划，并按现有单站点架构（`apps/web/`）落地，避免形成新的并行工程。

产出目标：

- 在现有站点中新增“项目案例页 + 可操作演示说明 + 风险边界说明”
- 给出后台能力接入策略（前端先行、后端可替换）
- 将“一键跑路”定义为合规的“站点降级/应急开关”机制，而非不可控破坏操作

## 2. 现状与约束

### 2.1 现状

- 当前仓库已收敛到单站点：`apps/web/`
- 文档治理已集中在 `docs/`，并有计划/执行/维护分层
- 发布链路走 GitHub Pages，前端静态发布稳定

### 2.2 约束

- 必须遵守“单一事实源”：站点能力只落在 `apps/web/`
- 规划写 `docs/plans/`，实施细节后续进入 `docs/architecture/`
- 不引入与当前架构冲突的第二套站点或第二套内容源

## 3. 整合范围（In Scope / Out of Scope）

### 3.1 In Scope

- 新增“案例整合专题”内容页（说明功能、交互、技术取舍）
- 新增后台能力的接口契约草案（配置读取、开关控制、审计日志）
- 新增“应急开关（原一键跑路）”的产品与技术规范
- 新增阶段性验收标准与发布回归清单

### 3.2 Out of Scope（本轮不做）

- 不直接上线破坏性删除功能
- 不引入复杂多租户后台系统
- 不改动现有发布主链路（Pages + Actions）

## 4. 功能模块拆分

### 模块 A：案例内容整合层（前台）

- A1：在 `apps/web/src/content/posts/` 新增案例文章
- A2：文章中拆分“前台展示 / 后台管理 / 应急开关”三段结构
- A3：补充操作截图位、演示链接位、版本记录位

交付物：可公开访问的案例页与更新日志条目。

### 模块 B：后台能力契约层（接口先行）

- B1：定义配置读取接口（站点信息、主题参数、联系方式、开关状态）
- B2：定义配置写入接口（字段校验、版本号、回滚点）
- B3：定义最小鉴权策略（单用户 Token 或管理员会话）

交付物：接口契约文档（先文档化，不强制本轮实现完整后端）。

### 模块 C：应急开关治理层（“一键跑路”重定义）

- C1：将“一键跑路”重命名为“应急开关 / Safe Mode”
- C2：定义 3 档策略：
  - Level 1：隐藏敏感入口（联系方式/外链）
  - Level 2：切换到只读维护页
  - Level 3：站点公告 + 暂停交互组件
- C3：每次切换必须留痕（操作者、时间、原因、前后状态）

交付物：可审计的开关策略文档与页面行为规范。

### 模块 D：发布与回归层

- D1：补充“案例页发布回归检查项”
- D2：补充“应急开关变更回归检查项”
- D3：将检查结论记录到维护日志

交付物：可复用的发布检查清单与复盘模板。

## 5. 分阶段推进计划

## 阶段 P0（规划固化，1-2 天）

目标：先把“整合边界与能力拆分”定稿。

任务：

- 在 `docs/plans/` 固化本计划
- 在 `docs/README.md`、`docs/plans/README.md` 挂载入口
- 明确“应急开关”命名与行为边界

验收标准：

- 规划文档可一跳访问
- 范围、模块、阶段、验收标准完整

## 阶段 P1（前台内容落地，2-4 天）

目标：完成可公开展示的案例整合页面。

任务：

- 新增案例文章与结构化章节
- 更新站点导航或聚合入口（如分类/系列/更新页）
- 增加更新日志条目（`public-data/updates/index.md`）

验收标准：

- 案例页可从站点入口访问
- 文章结构包含“功能说明 + 操作流程 + 风险声明”

## 阶段 P2（接口契约与开关规范，3-5 天）

目标：让后台能力具备可实施设计稿。

任务：

- 输出接口契约文档（读写配置 + 鉴权 + 审计）
- 输出应急开关状态机与 UI 行为映射
- 补齐异常场景处理（误触恢复、回滚、重复提交）

验收标准：

- 契约文档可指导后端实现
- 应急开关策略可被测试用例覆盖

## 阶段 P3（试运行与治理收口，持续）

目标：将该能力纳入长期治理。

任务：

- 将关键检查并入 `npm run check` 的治理路线（先文档后脚本）
- 在维护日志记录每次策略变更
- 建立最小回归快照（前后页面对照）

验收标准：

- 变更可追溯、可回滚、可审计
- 不影响现有发布稳定性

## 6. 目录落位建议

- 计划：`docs/plans/rainyun-topic-13009-integration-plan.md`
- 实施清单（后续）：`docs/architecture/` 下新增对应 checklist
- 维护记录（后续）：`docs/maintenance/astro-redesign-execution-log.md`
- 公开更新：`public-data/updates/index.md`
- 前台内容：`apps/web/src/content/posts/`

## 7. 风险与应对

- 风险 1：功能描述偏“噱头”，与当前站点调性冲突
  - 应对：先做案例化呈现，明确实验性质与治理边界
- 风险 2：应急开关被误用
  - 应对：双确认 + 操作日志 + 可回滚
- 风险 3：新增内容破坏现有信息架构
  - 应对：仅挂载到既有内容体系，不新开并行导航系统

## 8. 本计划执行口径

- 本文档是“整合计划”，不是立即上线全部能力的承诺。
- 先完成 P0/P1，再决定是否进入 P2 的接口实现。
- 所有落地动作遵循现有仓库分层：计划 -> 架构 -> 维护 -> 发布记录。
