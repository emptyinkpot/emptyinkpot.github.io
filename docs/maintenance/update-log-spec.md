# 博客更新日志治理规范

更新时间：2026-03-27

## 1. 目标

更新日志系统的目标不是“多写一篇说明文”，而是建立一个长期可维护的变更治理机制。

这套机制要同时满足：

- 可追溯：能从公开日志追到提交、文件、页面
- 可约束：每条记录字段一致，便于回顾和筛选
- 可自动化：新增和校验有标准命令
- 可复用：GitHub 更新说明、阶段回顾和对外说明可以复用同一份信息

## 2. 信息源分工

| 类型 | 位置 | 面向对象 | 用途 |
| --- | --- | --- | --- |
| 公开维护日志 | `public-data/updates/index.md` | 站点读者 | 说明博客更新了什么、效果是什么 |
| 内部执行日志 | `docs/maintenance/astro-redesign-execution-log.md` | 仓库维护者 | 记录阶段推进、验证、下一步 |
| 执行清单 | `docs/architecture/astro-blog-redesign-checklist.md` | 项目维护者 | 记录范围、阶段状态、待办 |
| Git 历史 | `git log` / `git show` | 仓库维护者 | 提供最细粒度的事实追踪 |

原则：

- 对用户可感知的变化，必须写进公开维护日志
- 对阶段推进与任务切换，必须更新执行清单或内部执行日志
- 公开日志不能代替 Git 历史，但要能指向 Git 历史

## 3. 每条公开日志的必填字段

每条条目必须按以下顺序书写：

1. `## YYYY-MM-DD / 标题`
2. `### 更新内容`
3. `### 涉及技术`
4. `### 关联记录`
5. `### 验证记录`
6. `### 实现效果`
7. `### 下一步`

补充要求：

- 每个区块至少有 1 条列表项
- `关联记录` 至少包含以下 2 项：
  - 页面 / URL
  - 文件 / 目录
  - 提交号
- `验证记录` 至少写 1 条真实执行过的验证

## 4. 标题与内容规范

标题规范：

- 使用绝对日期，不使用“今天”“昨天”
- 标题描述一次明确变化，不写“继续优化”“一些调整”这类模糊表达
- 优先使用“动作 + 对象”的表达，例如：
  - 首页改为文章列表，点击进入全文
  - 建立 `site-v2` Astro 原型站
  - 修复 Hexo 的 GitHub Pages 自动部署

内容规范：

- `更新内容` 写事实，不写感受
- `涉及技术` 写具体技术或配置，不写大而泛的术语
- `实现效果` 写结果，不重复过程
- `下一步` 只写紧接着要做的动作

## 5. 质量门槛

满足以下条件后，才算一条合格日志：

- 能从日志看出更新范围
- 能从日志看出技术点
- 能从日志追到文件或提交
- 能从日志看出至少 1 条验证事实
- 能知道下一步往哪推进

不合格示例：

- 只有“优化了一下样式”
- 只有“修了点问题”
- 没有写验证方式
- 没有关联文件或提交

## 6. 自动化要求

新增条目：

```bash
npm run new:update -- --title "更新标题" --dry-run
```

校验条目：

```bash
npm run check:updates
npm run check:content
npm run check
npm run build
```

建议固定验证顺序：

1. 完成功能或内容修改
2. 如需新增公开维护记录，先执行 `npm run new:update -- --title "更新标题" --dry-run` 生成草稿参考
3. 更新 `public-data/updates/index.md`
4. 再更新 `docs/maintenance/astro-redesign-execution-log.md` 与执行清单
5. 执行 `npm run check`，统一覆盖更新日志、内容治理与仓库治理校验
6. 如需单独排查内容 front matter 或历史 slug 兼容问题，可补充执行 `npm run check:content`
7. 执行 `npm run build`，确认 Astro 站点可构建
8. 提交并推送

## 7. 与 GitHub 更新说明的关系

推荐从公开更新日志中提炼 GitHub 更新说明。

优先复用以下字段：

- 更新内容
- 涉及技术
- 实现效果

必要时补充：

- 关联提交
- 验证记录
- 下一步
